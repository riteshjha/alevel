Algorithm Design & Problem-Solving

Paper 2 · AS & A Level ~15% of Paper 2 marks

  • Write correct Cambridge pseudocode using all required constructs

  • Describe, implement, and compare bubble, insertion, merge, and quick sort

1 Cambridge Pseudocode Conventions

Pseudocode is a structured, language-independent way of describing an algorithm. Cambridge 9618 uses a specific pseudocode style — you must follow it in Paper 2 answers.

Variables and Data Types

Cambridge Pseudocode // Declaration DECLARE Count : INTEGER DECLARE Price : REAL DECLARE Name : STRING DECLARE Pass : BOOLEAN DECLARE Grade : CHAR // Assignment uses ← (not =) Count ← 0 Price ← 9.99 Name ← "Alice" Pass ← TRUE // Arrays: 1-indexed by convention DECLARE Scores : ARRAY[1:10] OF INTEGER Scores[1] ← 85 // 2D array DECLARE Grid : ARRAY[1:3, 1:3] OF INTEGER

Input, Output, and Operators

Cambridge Pseudocode // Input and output INPUT Value OUTPUT "Enter your name: " OUTPUT "Score is: ", Score // Arithmetic operators // + - * / (standard) // DIV → integer division (truncates) // MOD → remainder Result ← 17 DIV 5 // Result = 3 Remain ← 17 MOD 5 // Remain = 2 // Comparison operators // = <> < > <= >= // Logical operators // AND OR NOT // String concatenation FullName ← FirstName & " " & LastName

Selection (IF and CASE)

Cambridge Pseudocode // IF … THEN … ELSE … ENDIF IF Score >= 50 THEN OUTPUT "Pass" ELSE OUTPUT "Fail" ENDIF // Nested IF IF Score >= 70 THEN OUTPUT "Distinction" ELSE IF Score >= 50 THEN OUTPUT "Pass" ELSE OUTPUT "Fail" ENDIF ENDIF // CASE OF … ENDCASE CASE OF Grade 'A' : OUTPUT "Excellent" 'B' : OUTPUT "Good" 'C' : OUTPUT "Satisfactory" OTHERWISE : OUTPUT "Below standard" ENDCASE

Iteration (Loops)

Cambridge Pseudocode // FOR loop — use when number of iterations is known FOR i ← 1 TO 10 Total ← Total + Scores[i] NEXT i // FOR with STEP FOR i ← 10 TO 1 STEP -1 OUTPUT i NEXT i // WHILE loop — condition checked before each iteration WHILE Count < 5 DO Count ← Count + 1 ENDWHILE // REPEAT…UNTIL — body always runs at least once REPEAT INPUT Value IF Value < 0 THEN OUTPUT "Must be positive" ENDIF UNTIL Value >= 0

Procedures and Functions

Cambridge Pseudocode // PROCEDURE — no return value PROCEDURE PrintHeading(Title : STRING) OUTPUT "=== " & Title & " ===" ENDPROCEDURE // Call a procedure CALL PrintHeading("Results") // FUNCTION — returns a value FUNCTION Max(A : INTEGER, B : INTEGER) RETURNS INTEGER IF A > B THEN RETURN A ELSE RETURN B ENDIF ENDFUNCTION // Call a function Largest ← Max(7, 12) // BYREF — changes to parameter affect original variable PROCEDURE Swap(BYREF X : INTEGER, BYREF Y : INTEGER) Temp ← X X ← Y Y ← Temp ENDPROCEDURE
Exam tip: Cambridge mark schemes penalise wrong pseudocode syntax. Use ← for assignment (not =), always close loops with NEXT/ENDWHILE/UNTIL/ENDFOR, and terminate PROCEDURE/FUNCTION with ENDPROCEDURE/ENDFUNCTION. Indentation is expected.

Trace Tables

A trace table is used to manually simulate an algorithm step-by-step, tracking the value of each variable at each step. It is a key exam technique.

Trace Table Example

Trace the following algorithm for the array [5, 3, 8, 1]:

FOR i ← 1 TO 3 IF Data[i] > Data[i+1] THEN CALL Swap(Data[i], Data[i+1]) ENDIF NEXT i
iData[1]Data[2]Data[3]Data[4]Swap?
Start5381—
13581Yes (5>3)
23581No (5<8)
33518Yes (8>1)

After one pass: [3, 5, 1, 8] — the largest element (8) has bubbled to the end.

2 Sorting Algorithms

Sorting places data into a defined order (ascending or descending). Cambridge 9618 requires you to describe, trace, implement in pseudocode, and compare the following algorithms.

Bubble Sort

O(n²)
  • Compare adjacent pairs, swap if out of order
  • Repeat passes until no swaps occur
  • Each pass bubbles the next largest to its place
  • Simple to implement; very inefficient for large data
  • Stable sort

Insertion Sort

O(n²)
  • Build a sorted sub-array left to right
  • Insert each unsorted element into correct position
  • Efficient for small or nearly-sorted data
  • O(n) best case on already-sorted input
  • Stable sort

Merge Sort

O(n log n)
  • Divide array into two halves recursively
  • Sort each half, then merge sorted halves
  • Consistent O(n log n) regardless of input
  • Requires O(n) extra memory
  • Stable sort; well-suited to large datasets

Quick Sort

O(n log n) avg
  • Choose a pivot; partition into <pivot and >pivot
  • Recursively sort each partition
  • Fast in practice; in-place (O(log n) stack space)
  • Worst case O(n²) with bad pivot choices
  • Not stable (relative order may change)

Bubble Sort — Pseudocode

Cambridge Pseudocode PROCEDURE BubbleSort(BYREF Data : ARRAY[1:100] OF INTEGER, n : INTEGER) DECLARE Swapped : BOOLEAN DECLARE Temp : INTEGER REPEAT Swapped ← FALSE FOR i ← 1 TO n - 1 IF Data[i] > Data[i + 1] THEN Temp ← Data[i] Data[i] ← Data[i + 1] Data[i + 1] ← Temp Swapped ← TRUE ENDIF NEXT i UNTIL NOT Swapped ENDPROCEDURE

Insertion Sort — Pseudocode

Cambridge Pseudocode PROCEDURE InsertionSort(BYREF Data : ARRAY[1:100] OF INTEGER, n : INTEGER) DECLARE Key : INTEGER DECLARE j : INTEGER FOR i ← 2 TO n Key ← Data[i] j ← i - 1 WHILE j >= 1 AND Data[j] > Key DO Data[j + 1] ← Data[j] j ← j - 1 ENDWHILE Data[j + 1] ← Key NEXT i ENDPROCEDURE

Merge Sort — Concept

Merge sort uses the divide and conquer strategy: split in half until each sub-array has one element (trivially sorted), then merge pairs back in sorted order.

Merge Sort Trace — [38, 27, 43, 3]
1
Split: [38, 27, 43, 3] → [38, 27] and [43, 3]
2
Split again: [38, 27] → [38] and [27]    [43, 3] → [43] and [3]
3
Merge [38] and [27]: compare 38 vs 27 → [27, 38]
4
Merge [43] and [3]: compare 43 vs 3 → [3, 43]
5
Merge [27, 38] and [3, 43]: 3<27 → take 3; 27<43 → take 27; 38<43 → take 38; take 43 → [3, 27, 38, 43]

Quick Sort — Concept

Quick sort chooses a pivot element, partitions the array so all elements less than the pivot come before it and all greater elements come after it, then recursively sorts each partition.

Quick Sort Trace — [5, 3, 8, 1, 9, 2], pivot = first element
1
Pivot = 5. Partition: [3, 1, 2] | 5 | [8, 9]
2
Recursively sort [3, 1, 2]: pivot = 3 → [1, 2] | 3 → sort [1, 2] → [1, 2] | 3 → [1, 2, 3]
3
Recursively sort [8, 9]: pivot = 8 → 8 | [9] → [8, 9]
4
Combine: [1, 2, 3, 5, 8, 9]

Sorting Algorithm Comparison

AlgorithmBest CaseAverage CaseWorst CaseSpaceStable?
Bubble SortO(n)O(n²)O(n²)O(1)Yes
Insertion SortO(n)O(n²)O(n²)O(1)Yes
Merge SortO(n log n)O(n log n)O(n log n)O(n)Yes
Quick SortO(n log n)O(n log n)O(n²)O(log n)No
Choosing a sort: For small or nearly-sorted data, insertion sort is efficient and simple. For large datasets where consistent performance matters, use merge sort. Quick sort is fastest in practice on random data due to low constant factors, but avoid it when worst-case performance is critical.

3 Searching Algorithms

Searching finds whether a target value exists in a collection and, if so, returns its location.

Linear Search — O(n)

Check each element in sequence from the beginning. Works on unsorted or sorted data.

Cambridge Pseudocode FUNCTION LinearSearch(Data : ARRAY[1:100] OF INTEGER, n : INTEGER, Target : INTEGER) RETURNS INTEGER // Returns index of Target, or -1 if not found FOR i ← 1 TO n IF Data[i] = Target THEN RETURN i ENDIF NEXT i RETURN -1 ENDFUNCTION
Best CaseWorst CaseAverage Case
Linear SearchO(1) — first element is targetO(n) — target at end or absentO(n/2) ≈ O(n)

Binary Search — O(log n)

Requires sorted data. Repeatedly halves the search space by comparing the target to the middle element.

Cambridge Pseudocode FUNCTION BinarySearch(Data : ARRAY[1:100] OF INTEGER, n : INTEGER, Target : INTEGER) RETURNS INTEGER DECLARE Low, High, Mid : INTEGER Low ← 1 High ← n WHILE Low <= High DO Mid ← (Low + High) DIV 2 IF Data[Mid] = Target THEN RETURN Mid ELSE IF Data[Mid] < Target THEN Low ← Mid + 1 // target in upper half ELSE High ← Mid - 1 // target in lower half ENDIF ENDIF ENDWHILE RETURN -1 // not found ENDFUNCTION
Binary Search Trace — Target = 23 in [5, 9, 12, 17, 23, 31, 45]
StepLowHighMidData[Mid]Action
11741717 < 23 → Low = 5
25763131 > 23 → High = 5
355523Found at index 5

Binary search halved the search space each step — only 3 comparisons for 7 elements. Linear search could have needed up to 7.

Searching Algorithm Comparison

FeatureLinear SearchBinary Search
Requires sorted data?NoYes

Worst caseO(n)O(log n)
Suitable for large datasets?No — too slowYes — very efficient

Exam tip: If asked which search to use, state the precondition — binary search requires sorted data. If data is not sorted, either sort first (at a cost) or use linear search. Cambridge questions often ask you to justify your choice.

4 Algorithm Complexity & Big-O Notation

Big-O notation describes how the running time (or memory usage) of an algorithm grows as the input size n increases. It expresses the worst-case growth rate, ignoring constants and lower-order terms.

Common Complexity Classes

NotationNameGrowthExample
O(1)ConstantSame time regardless of nAccessing an array element by index
O(log n)LogarithmicDoubles input → one extra stepBinary search
O(n)LinearDoubles input → doubles timeLinear search, reading all n items
O(n log n)LinearithmicSlightly worse than linearMerge sort, quick sort (average)
O(n²)QuadraticDoubles input → quadruples timeBubble sort, insertion sort, nested loops
O(2ⁿ)ExponentialAdding one element → doubles timeNaive recursive Fibonacci, brute-force TSP

Identifying Complexity from Code Structure

Code StructureTypical Complexity
Single statement / no loopO(1)
One loop over n elementsO(n)
Two nested loops over n elementsO(n²)
Halving the problem each iteration (e.g. binary search)O(log n)
Loop + recursive halving (e.g. merge sort)O(n log n)
Big-O simplification: Drop constants and lower-order terms. O(3n² + 2n + 7) simplifies to O(n²) — we only care about the dominant term as n grows large.

Tractable vs Intractable Problems

An algorithm is tractable if it can be solved in polynomial time — O(nᵏ) for some constant k. Problems that have no known polynomial-time solution are called intractable.

CategoryDefinitionExample
TractableSolvable in polynomial time; feasible for large inputsSorting, searching, shortest path
IntractableNo known polynomial algorithm; only feasible for very small inputsTravelling Salesman (exact), graph colouring
Heuristic solutionApproximation that gives a "good enough" answer quicklyNearest-neighbour for TSP
Heuristics: When an exact solution is intractable, a heuristic trades guaranteed optimality for speed. The solution found may not be the best possible, but it is found in reasonable time. Example: for the Travelling Salesman Problem, always travel to the nearest unvisited city — not optimal, but fast.

5 Problem-Solving Strategies

Before writing an algorithm, you must understand and decompose the problem. Cambridge 9618 expects you to apply the following computational thinking skills.

Computational Thinking Components

ComponentDefinitionExample
DecompositionBreak a complex problem into smaller, more manageable sub-problemsA school management system broken into: student records, timetabling, reporting
AbstractionFocus on essential details; ignore irrelevant onesModelling a car as speed, fuel level, direction — ignoring paint colour
Pattern RecognitionIdentify similarities between problems or within dataRecognising that search appears in many sub-tasks
Algorithm DesignDefine step-by-step instructions to solve the problemWriting pseudocode for a sort or search

Top-Down Design (Stepwise Refinement)

Top-down design starts with the overall problem and repeatedly breaks it into smaller sub-tasks until each sub-task is simple enough to code directly. The result is a hierarchy of modules.

Top-Down Design — Student Grade System
1
Top level: Run Grade System
2
Level 2: Input Scores | Calculate Grades | Output Report
3
Input Scores → Level 3: Read number of students | For each student: validate and store score
4
Calculate Grades → Level 3: For each score: apply grade boundary | Store grade
5
Output Report → Level 3: Print header | Print each student's name and grade | Print statistics

Structure Charts

A structure chart shows the hierarchy of modules in a top-down design. Each box is a module; lines show which module calls which. Parameters passed between modules are labelled on the connecting lines.

Exam tip: When asked to design a solution, always show decomposition first — list the sub-tasks before writing pseudocode. Cambridge mark schemes award marks for a structured approach, not just a working solution. Show your thinking explicitly.

TechniquePurposeExample

Practice Question 3 — 6 marks

Write pseudocode for a function FindMin that takes an array of n integers and returns the smallest value. Then state its time complexity and explain why.

Cambridge Pseudocode FUNCTION FindMin(Data : ARRAY[1:100] OF INTEGER, n : INTEGER) RETURNS INTEGER DECLARE MinVal : INTEGER MinVal ← Data[1] FOR i ← 2 TO n IF Data[i] < MinVal THEN MinVal ← Data[i] ENDIF NEXT i RETURN MinVal ENDFUNCTION

Time complexity: O(n). The algorithm must examine every element exactly once — it cannot stop early because the minimum could be anywhere in an unsorted array. The loop runs n–1 times, giving linear O(n) complexity.

Practice Question 4 — 4 marks

State the time complexity of each algorithm below and briefly justify your answer:

  1. A loop that prints every element of an array of n items
  2. Two nested loops each running n times
  3. Binary search on a sorted list of n items
  4. Merge sort on n items
  • O(n) — the loop runs exactly n times; time is proportional to input size.
  • O(n²) — the inner loop runs n times for each of the n outer iterations: n × n = n² comparisons.
  • O(log n) — each comparison halves the remaining search space; the number of steps is log₂(n).
  • O(n log n) — the array is split log n times (divide phase), and each merge level processes all n elements (merge phase), giving n × log n total operations.
Practice Question 5 — 5 marks

A teacher wants to find a student's record in a list of 1 024 students sorted alphabetically by surname. Compare a linear search and a binary search in terms of the maximum number of comparisons needed, and justify which is more appropriate here.

Linear search: In the worst case (student not found or at the very end), all 1 024 records must be checked → 1 024 comparisons.

Binary search: Each step halves the remaining list. log₂(1 024) = 10, so at most 10 comparisons are needed.

Binary search is more appropriate because: (1) the list is sorted — the required precondition is met; (2) it requires only 10 comparisons versus up to 1 024 for linear search; (3) the efficiency advantage grows further as the list grows. The cost of maintaining sorted order is justified when many searches are performed.

Common Mistakes

  • Using = instead of ← for assignment
  • Forgetting ENDWHILE, ENDIF, ENDPROCEDURE
  • Binary search on unsorted data
  • Saying bubble sort is O(n) — it's O(n) best case only
  • Confusing DIV (integer division) with / (real division)
  • Off-by-one errors in loop bounds — Cambridge arrays are 1-indexed
