Software Development

The Software Development Life Cycle, analysis and design methods, testing strategies, test data categories, debugging techniques, and documentation — everything needed to produce and maintain reliable software.

Paper 2 · AS & A Level ~15% of Paper 2 marks

✦ Learning Objectives

By the end of this topic you should be able to:

  • Name and describe all stages of the Software Development Life Cycle
  • Compare waterfall and iterative/agile development models
  • Identify inputs, outputs, and processes during the analysis stage
  • Produce and interpret structure charts and data flow diagrams
  • Distinguish white-box, black-box, alpha, beta, and acceptance testing
  • Construct a test plan with normal, boundary, and erroneous test data
  • Apply trace tables to trace and debug code
  • Describe dry runs and the use of debugging tools
  • Explain the purpose of user and technical documentation
  • Distinguish adaptive, perfective, and corrective maintenance

1 The Software Development Life Cycle (SDLC)

The Software Development Life Cycle (SDLC) is a structured process that guides how software is planned, created, tested, and maintained. It provides a systematic approach that reduces errors, controls costs, and ensures the final product meets user needs.

Stage 1

Analysis

Identify the problem, define requirements, and document what the system must do. Produces a requirements specification.

Stage 2

Design

Plan the solution structure — data structures, algorithms, interfaces, and system architecture — before writing any code.

Stage 3

Coding

Translate the design into actual program code using an appropriate language and following coding standards.

Stage 4

Testing

Verify the program works correctly, handles errors, and meets the original requirements. Uses a structured test plan.

Stage 5

Implementation

Deploy the software to users. Includes installation, data migration, staff training, and system changeover.

Stage 6

Maintenance

Fix bugs that appear after release, adapt to changing requirements, and improve performance over time.

Development Models

Waterfall Model

Each stage is completed fully before the next begins. The flow is strictly sequential — like water falling over a series of steps. Changes late in the process are expensive because earlier stages must be revisited.

AdvantageDisadvantage
Clear documentation at each stageInflexible — hard to go back to a previous stage
Easy to manage and estimate timescalesUsers don't see the product until the end
Works well when requirements are fixed from the startLate discovery of errors is very costly to fix

Iterative / Agile Model

Development happens in short repeated iterations (sprints). Each iteration delivers a working subset of features. User feedback shapes the next iteration, allowing requirements to evolve throughout the project.

AdvantageDisadvantage
Users see and test working software earlyHarder to predict overall cost and timeline
Flexible — requirements can change between iterationsRequires continuous user involvement
Errors discovered and fixed quicklyDocumentation can lag behind development
Exam tip: You may be asked to recommend a model for a given scenario. Choose waterfall when requirements are fully defined upfront (e.g., safety-critical systems). Choose iterative when requirements are likely to evolve or when early user feedback is valuable.

2 Analysis & Design

Analysis Stage

The goal of analysis is to understand the current system (if one exists) and precisely define what the new system must achieve. Techniques include:

  • Interviewing stakeholders — ask users and managers what they need
  • Questionnaires — gather information from many users efficiently
  • Observation — watch how the current system is used in practice
  • Examining existing documentation — review current forms, reports, and processes

The output is a requirements specification listing both:

  • Functional requirements — what the system must do (e.g., "the system must calculate the total price including VAT")
  • Non-functional requirements — constraints on how it works (e.g., response time under 2 seconds, 99.9% uptime)

Structure Charts

A structure chart (also called a hierarchy chart) shows the modular decomposition of a system — how a large task is broken into smaller sub-tasks, and how those sub-tasks call each other. Each box represents a module (procedure or function).

Data Flow Diagrams (DFDs)

A Data Flow Diagram (DFD) shows how data moves through a system. It uses four symbols:

SymbolNameDescription
RectangleExternal EntityAn entity outside the system that sends or receives data (e.g., a customer, a bank)
Rounded rectangle / circleProcessA transformation of data — labelled with a verb phrase (e.g., "Validate login")
Open-ended rectangleData StoreA repository where data is held (e.g., a database table, a file)
ArrowData FlowThe direction data moves; labelled with what data is flowing
Remember: DFDs show data flow, not control flow. They do not show decisions or loops — those belong in flowcharts or pseudocode.

Feasibility Study

Before committing to a project, a feasibility study assesses whether the proposed system is worth building. It considers:

  • Technical feasibility — can the technology required actually be built?
  • Economic feasibility — will the benefits justify the cost?
  • Time feasibility — can it be delivered within the required timescale?
  • Legal feasibility — does the system comply with relevant laws (e.g., GDPR)?

3 Testing Strategies

Testing verifies that software behaves correctly under all expected (and unexpected) conditions. A good test plan covers all paths through the code and all categories of input.

White-Box vs Black-Box Testing

White-Box Testing

The tester has full knowledge of the internal code structure. Tests are designed to exercise every statement, branch, and loop path. Also called structural or glass-box testing.

Black-Box Testing

The tester has no knowledge of the internal code. Tests are based purely on the specification — inputs and expected outputs. Also called functional or specification-based testing.

Alpha Testing

Carried out by the development team (or in-house testers) before the software is released externally. Typically white-box or structured black-box testing in a controlled environment.

Beta Testing

The software is released to a limited group of real external users who use it in their normal environment and report bugs. Uncontrolled — testers follow their own usage patterns.

Acceptance Testing

Performed by the client to confirm the system meets the requirements specification. Also called User Acceptance Testing (UAT). Passes sign-off before go-live.

Integration Testing

Tests that separately developed modules work correctly together when combined into the larger system. Finds interface mismatches between modules.

Unit Testing

Tests individual modules (functions or procedures) in isolation to verify each one works correctly on its own. Usually performed by the programmer who wrote the unit.

Regression Testing

Re-running previously passed tests after a change to ensure new code has not broken existing functionality. Critical during iterative development.

4 Test Data

Choosing the right test data is essential for thorough testing. Cambridge requires three categories:

Category Definition Purpose
Normal Valid data within the accepted range Confirms the system produces correct output for typical inputs
Boundary Values at the exact edges of the accepted range (and one either side) Tests the limits of validation logic — common site of off-by-one errors
Erroneous Invalid data — wrong type, out of range, or deliberately incorrect Confirms the system rejects bad input and handles errors gracefully
Note: Some syllabi also list extreme data (valid but at the very edge of what the system can handle — e.g., an empty list, or exactly 1000 records). Cambridge 9618 primarily uses Normal / Boundary / Erroneous.

Worked Example: Age Validation (16 ≤ age ≤ 65)

Writing a Test Plan in Pseudocode Context

When a question gives you a function and asks for a test plan, always include:

  • At least one normal value (mid-range)
  • The exact boundary values and those immediately outside
  • At least one erroneous value (wrong type or clearly invalid)
  • The expected output — not just "pass" or "fail"

5 Debugging

Debugging is the process of finding and fixing errors (bugs) in a program. There are three main types of error:

Error TypeWhen DetectedDescriptionExample
Syntax Error Compile time / translation Code that violates the grammar rules of the language. The translator cannot convert it to machine code. IF x = 5 without THEN in pseudocode
Runtime Error During execution Code is syntactically correct but causes a crash or exception when it runs (e.g., division by zero, accessing an array out of bounds). Dividing by a variable that happens to be 0
Logic Error During testing Code runs without crashing but produces incorrect results. The algorithm is wrong. These are the hardest to find. Using > instead of >= in a boundary check

Dry Run (Trace Table)

A dry run means manually stepping through code line-by-line without running it on a computer. A trace table records the value of each variable at each step — it is the standard way to show a dry run in an exam.

Debugging Tools

Modern IDEs provide tools that assist in debugging:

ToolDescription
BreakpointPauses execution at a specific line so the programmer can inspect the current state of all variables
Step (Step Over / Step Into)Execute one line at a time; step into enters a called function, step over treats it as a single step
Watch / Variable InspectorDisplays the current value of selected variables, updating as execution proceeds
Call Stack ViewerShows which functions have been called and in what order — essential for debugging recursion
Error / Exception MessagesRuntime error messages from the interpreter/compiler indicating the type of error and the line number
Exam tip: "Describe how a programmer could find a logic error" — a common answer is: use a dry run / trace table, insert OUTPUT statements to print variable values at key points, or use an IDE debugger with breakpoints and a variable watch.
Practice Question

A programmer writes a function to find the maximum value in an array. During testing, the function always returns the first element. Name the type of error and describe two methods the programmer could use to find and fix it.

Type of error: Logic error — the program runs without crashing but produces an incorrect result.

Method 1: Dry run / trace table — manually trace through the function with a small test array, recording the value of the loop counter, current element, and current maximum at each step to identify where the logic fails.

Method 2: Use an IDE debugger — set a breakpoint inside the loop, then use step-through execution and a variable watch to observe the maximum variable updating (or failing to update) on each iteration.

6 Documentation & Maintenance

User Documentation

User documentation is written for the end-users of the software — people who use the system but may have no programming knowledge. Its purpose is to help users operate the software correctly and efficiently.

User documentation typically includes:

  • Getting started guide / installation instructions
  • Step-by-step instructions for common tasks with screenshots
  • Explanation of all menus, buttons, and input fields
  • Troubleshooting / FAQ section for common problems
  • Glossary of terms used in the software

Technical Documentation

Technical documentation is written for developers and system administrators who need to maintain, extend, or support the software. It assumes programming knowledge.

Technical documentation typically includes:

  • System requirements and dependencies
  • Installation and configuration guide for developers
  • Annotated source code explaining algorithms and non-obvious logic
  • Data dictionary describing all variables, data structures, and file formats
  • Entity-relationship diagrams, structure charts, data flow diagrams
  • API documentation — descriptions of all modules, their parameters, and return values
  • Known issues and limitations

AudienceEnd usersDevelopers / administrators
Technical levelNon-technical language, lots of screenshotsTechnical language, code listings, diagrams

Software Maintenance

After a system is deployed, it will need ongoing maintenance. There are three types:

TypeDescriptionExample
Corrective Fixing bugs that were not caught during testing and are discovered after release A bank calculates interest incorrectly due to a logic error in the rate formula
Adaptive Modifying the system to work in a new environment or to meet changing external requirements Updating the software to comply with new GDPR regulations, or to run on a new operating system
Perfective Improving the system's performance or adding new features requested by users, without fixing a specific bug Adding a search filter feature that users have requested, or optimising a slow database query
Exam tip: Given a scenario, identify the maintenance type: corrective = fix a known bug; adaptive = respond to external change (law, OS, hardware); perfective = user wants new/improved functionality.

System Changeover Methods

When replacing an old system with a new one, the organisation must choose how to switch over:

MethodDescriptionRisk
Direct changeover Old system is switched off and new system is switched on simultaneously High — if the new system fails there is no fallback
Parallel running Both old and new systems run together; outputs are compared until confidence is established Low — but expensive (double the resources)
Pilot changeover New system is introduced in one part of the organisation (e.g., one branch) first Medium — limited exposure if the new system fails
Phased changeover The new system replaces the old one module by module, one piece at a time Low — but requires both systems to coexist during transition

📝 Exam Practice

Question 1 — 4 marks

State four stages of the Software Development Life Cycle and for each stage, give one task that takes place during it.

Any four of the following (one mark per stage + task pair):

  • Analysis — gather and document user requirements (requirements specification)
  • Design — produce structure charts / data flow diagrams / algorithm design
  • Coding — write the program code in a chosen programming language
  • Testing — run the program against a test plan with normal, boundary, and erroneous data
  • Implementation — install the system and train users
  • Maintenance — fix bugs and adapt the system to new requirements
Question 2 — 3 marks

Explain the difference between white-box and black-box testing. For each, give one example of when it would be used.

White-box testing: The tester has knowledge of the internal code structure and designs tests to exercise every path through the code. Used during unit testing by the developer to ensure every branch is executed at least once. (1 mark)

Black-box testing: The tester has no knowledge of the internal code; tests are based solely on the requirements specification — given these inputs, expect these outputs. Used during acceptance testing by the client to verify the system meets the agreed requirements. (1 mark)

Key difference: White-box is based on code structure; black-box is based on the specification. (1 mark)

Question 4 — 4 marks

A school's student records system is being replaced. Explain the difference between parallel running and direct changeover, and justify which method would be more appropriate for this system.

Direct changeover: The old system is switched off and the new system is switched on at the same time. If the new system fails, there is no fallback — this is high risk. (1 mark)

Parallel running: Both old and new systems operate simultaneously. Outputs from both are compared. Once confidence in the new system is established, the old system is retired. Lower risk but more expensive due to duplicated work. (1 mark)

Recommendation: Parallel running is more appropriate for a student records system because the data (student records, grades) is critical. If the new system has errors, the old system acts as a reliable fallback, preventing loss of important data. The extra cost is justified by the reduced risk. (2 marks)

Question 5 — 3 marks

A hospital's billing software was updated to comply with new government tax regulations. Six months later, users report that a specific discount calculation produces the wrong result. A year after that, users request a new feature to generate monthly summary reports.

For each of the three situations, identify the type of maintenance being carried out.

  1. Complying with new tax regulations → Adaptive maintenance — responding to an external change in requirements (legislation). (1 mark)
  2. Fixing the discount calculation bug → Corrective maintenance — fixing a fault discovered after release. (1 mark)
  3. Adding monthly summary reports → Perfective maintenance — adding new functionality requested by users to improve the system. (1 mark)

Common Mistakes

  • Forgetting the value one outside each boundary, not just the boundary values themselves
  • Confusing alpha (internal) and beta (external user) testing
  • Calling adaptive maintenance "corrective" — adaptive is about external change, not bugs
  • Writing logic errors as "syntax errors" — logic errors only appear in output, not during compilation
  • DFDs show data flow, not control flow — no loops or decisions
