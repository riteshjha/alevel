Data Types & Structures

Primitive data types, arrays, and the key abstract data types: stacks, queues, linked lists, binary trees, and hash tables — with pseudocode implementations and worked examples.

Paper 2 · AS & A Level ~15% of Paper 2 marks

1 Primitive Data Types

A primitive data type is a basic data type built into the language. In Cambridge pseudocode, all variables must be declared with their type before use.

TypeDescriptionStorageExample values
INTEGERWhole number (positive, negative, or zero)Typically 32 or 64 bits0, 42, -7, 1000
REALNumber with a fractional part (floating point)Typically 64 bits (double)3.14, -0.5, 9.99
CHARSingle character, enclosed in single quotes1 byte (ASCII) or 2 bytes (Unicode)'A', 'z', '9', ' '
STRINGSequence of zero or more charactersVariable length"Hello", "CS 9618", ""

Type Casting

Cambridge pseudocode uses built-in functions to convert between types:

Cambridge Pseudocode // Convert types Num ← INT(3.9) // Num = 3 (truncates, does not round) R ← REAL(7) // R = 7.0 S ← NUM_TO_STR(42) // S = "42" N ← STR_TO_NUM("42") // N = 42 // String functions L ← LENGTH("Hello") // L = 5 Sub ← SUBSTRING("Hello", 1, 3) // Sub = "Hel" U ← UCASE("hello") // U = "HELLO"
Exam tip: Know the difference between CHAR (single quotes, one character) and STRING (double quotes, any length). Mixing them in pseudocode is a common error that loses marks. Also remember INT() truncates — INT(3.9) = 3, not 4.

2 Arrays

An array is a fixed-size, ordered collection of elements all of the same type, stored in contiguous memory locations. Cambridge arrays are 1-indexed by convention.

1D Arrays

Cambridge Pseudocode // Declaration: ARRAY[lower:upper] OF type DECLARE Scores : ARRAY[1:5] OF INTEGER // Assign values Scores[1] ← 85 Scores[2] ← 72 Scores[3] ← 91 Scores[4] ← 68 Scores[5] ← 77 // Iterate through array Total ← 0 FOR i ← 1 TO 5 Total ← Total + Scores[i] NEXT i Average ← Total / 5

2D Arrays

A 2D array is a grid of rows and columns. It is declared as ARRAY[row1:row2, col1:col2]. Elements are accessed as Grid[row, col].

Cambridge Pseudocode // 3×4 grid (3 rows, 4 columns) DECLARE Grid : ARRAY[1:3, 1:4] OF INTEGER // Assign value at row 2, column 3 Grid[2, 3] ← 42 // Process all elements — nested loops FOR row ← 1 TO 3 FOR col ← 1 TO 4 Grid[row, col] ← row * col NEXT col NEXT row
Fixed size: Arrays in pseudocode have a fixed size declared at compile time. You cannot resize them at runtime. If you need a dynamic structure, use a linked list instead.

3 Stacks

A stack is a Last In, First Out (LIFO) abstract data type. Elements are added and removed from the same end — the top of the stack. Think of a stack of plates: you always add to and take from the top.

Stack Operations

OperationDescriptionError condition
Push(item)Add an item to the top of the stackStack overflow — stack is full
Pop()Remove and return the top itemStack underflow — stack is empty
Peek() / Top()Return the top item without removing itStack is empty
isEmpty()Return TRUE if the stack contains no items—
isFull()Return TRUE if the stack is at maximum capacity—

Stack Visualisation

After pushing 10, 20, 30 in that order (TopOfStack pointer = 3):

10
20
30 ← Top

Push(40): TopOfStack → 4, Stack[4] = 40

Pop(): returns 30, TopOfStack → 2

Peek(): returns Stack[TopOfStack] without changing TopOfStack

Stack Implementation using an Array

Cambridge Pseudocode // Stack stored in array Stack[1..MaxSize], TopOfStack tracks top position DECLARE Stack : ARRAY[1:100] OF INTEGER DECLARE TopOfStack : INTEGER DECLARE MaxSize : INTEGER MaxSize ← 100 TopOfStack ← 0 // 0 means empty PROCEDURE Push(Item : INTEGER) IF TopOfStack = MaxSize THEN OUTPUT "Error: Stack overflow" ELSE TopOfStack ← TopOfStack + 1 Stack[TopOfStack] ← Item ENDIF ENDPROCEDURE FUNCTION Pop() RETURNS INTEGER IF TopOfStack = 0 THEN OUTPUT "Error: Stack underflow" RETURN -1 ELSE Item ← Stack[TopOfStack] TopOfStack ← TopOfStack - 1 RETURN Item ENDIF ENDFUNCTION FUNCTION Peek() RETURNS INTEGER IF TopOfStack = 0 THEN OUTPUT "Error: Stack is empty" RETURN -1 ELSE RETURN Stack[TopOfStack] ENDIF ENDFUNCTION FUNCTION isEmpty() RETURNS BOOLEAN RETURN TopOfStack = 0 ENDFUNCTION FUNCTION isFull() RETURNS BOOLEAN RETURN TopOfStack = MaxSize ENDFUNCTION

Applications of Stacks

  • Call stack: When a function is called, its return address and local variables are pushed. When it returns, they are popped — this is how recursion works.
  • Undo/redo: Each action is pushed onto an undo stack. Ctrl+Z pops the last action and reverses it.
  • Expression evaluation: Operators and operands are pushed/popped to evaluate arithmetic expressions (e.g. converting infix to postfix notation).
  • Bracket matching: Open brackets are pushed; when a closing bracket is seen, the top is popped and checked for a match.

4 Queues

A queue is a First In, First Out (FIFO) abstract data type. Items are added at the back (rear) and removed from the front (head). Think of a queue of people: the first person to join is the first to be served.

Queue Operations

OperationDescriptionError condition
Enqueue(item)Add an item to the back of the queueQueue is full
Dequeue()Remove and return the item at the frontQueue is empty
Peek()Return the front item without removing itQueue is empty
isEmpty()Return TRUE if the queue contains no items—
isFull()Return TRUE if the queue is at maximum capacity—

Linear Queue — Problem

In a simple array-based queue, the Head pointer moves right as items are dequeued. This wastes the freed spaces at the front — the queue appears full even when most positions are empty.

After 2 dequeues:
—
—
30Head
40
50Tail
—

Indices 1–2 are wasted. The queue will report "full" when Tail reaches the end, despite having free space.

Circular Queue — Solution

A circular queue wraps the Tail pointer back to index 1 when it reaches the end of the array, reusing freed front positions. The next index after MaxSize is 1 (using MOD arithmetic).

Cambridge Pseudocode DECLARE Queue : ARRAY[1:6] OF INTEGER DECLARE Head : INTEGER DECLARE Tail : INTEGER DECLARE Count : INTEGER DECLARE MaxSize : INTEGER Head ← 1 Tail ← 0 Count ← 0 MaxSize ← 6 PROCEDURE Enqueue(Item : INTEGER) IF Count = MaxSize THEN OUTPUT "Error: Queue is full" ELSE Tail ← (Tail MOD MaxSize) + 1 Queue[Tail] ← Item Count ← Count + 1 ENDIF ENDPROCEDURE FUNCTION Dequeue() RETURNS INTEGER IF Count = 0 THEN OUTPUT "Error: Queue is empty" RETURN -1 ELSE Item ← Queue[Head] Head ← (Head MOD MaxSize) + 1 Count ← Count - 1 RETURN Item ENDIF ENDFUNCTION
Why Count? Using a Count variable (rather than checking Head = Tail) avoids ambiguity: if Head = Tail, is the queue empty or full? Count makes the distinction explicit and unambiguous.

Applications of Queues

  • Print spooling: Documents are queued for printing in the order they are submitted
  • CPU scheduling: Processes waiting for CPU time are held in a ready queue
  • Keyboard buffer: Keystrokes are queued so none are missed if the CPU is busy
  • Breadth-first search: Nodes to visit next are stored in a queue

Traversal

Cambridge Pseudocode // Linked list stored in parallel arrays: Data[] and Next[] // Head holds the index of the first node; 0 = NIL DECLARE Data : ARRAY[1:100] OF INTEGER DECLARE Next : ARRAY[1:100] OF INTEGER DECLARE Head : INTEGER PROCEDURE Traverse() Current ← Head WHILE Current <> 0 DO OUTPUT Data[Current] Current ← Next[Current] ENDWHILE ENDPROCEDURE

Insertion at Head

Cambridge Pseudocode // FreeList holds index of next available slot PROCEDURE InsertAtHead(NewData : INTEGER) IF FreeList = 0 THEN OUTPUT "Error: List is full" ELSE NewNode ← FreeList FreeList ← Next[FreeList] Data[NewNode] ← NewData Next[NewNode] ← Head Head ← NewNode ENDIF ENDPROCEDURE

Deletion

Cambridge Pseudocode PROCEDURE Delete(Target : INTEGER) // Find node before the one to delete IF Data[Head] = Target THEN // Delete head node Temp ← Head Head ← Next[Head] Next[Temp] ← FreeList FreeList ← Temp ELSE Prev ← Head WHILE Next[Prev] <> 0 AND Data[Next[Prev]] <> Target DO Prev ← Next[Prev] ENDWHILE IF Next[Prev] <> 0 THEN Temp ← Next[Prev] Next[Prev] ← Next[Temp] Next[Temp] ← FreeList FreeList ← Temp ENDIF ENDIF ENDPROCEDURE

6 Binary Trees

A binary tree is a hierarchical data structure where each node has at most two children: a left child and a right child. A Binary Search Tree (BST) imposes an ordering rule: left child < parent < right child.

Key Terminology

TermDefinition
RootThe topmost node — the starting point for all traversals
LeafA node with no children
Parent / ChildA node directly above / below another in the hierarchy
SubtreeA node and all its descendants
HeightNumber of edges on the longest path from root to a leaf

Building a BST — Worked Example

Insert the values 50, 30, 70, 20, 40, 60, 80 into an empty BST in that order:

BST Insertion
1
Insert 50 → becomes the root.
2
Insert 30 → 30 < 50, goes left of 50.
3
Insert 70 → 70 > 50, goes right of 50.
4
Insert 20 → 20 < 50, go left → 20 < 30, goes left of 30.
5
Insert 40 → 40 < 50, go left → 40 > 30, goes right of 30.
6
Insert 60 → 60 > 50, go right → 60 < 70, goes left of 70.
7
Insert 80 → 80 > 50, go right → 80 > 70, goes right of 70.
              50
            /    \
          30      70
         /  \    /  \
        20  40  60  80

BST Search and Insertion — Pseudocode

Cambridge Pseudocode // BST in parallel arrays: NodeData[], LeftPtr[], RightPtr[] // Root holds index of root node; 0 = no child (NIL) FUNCTION Search(Current : INTEGER, Target : INTEGER) RETURNS BOOLEAN IF Current = 0 THEN RETURN FALSE ELSE IF NodeData[Current] = Target THEN RETURN TRUE ELSE IF Target < NodeData[Current] THEN RETURN Search(LeftPtr[Current], Target) ELSE RETURN Search(RightPtr[Current], Target) ENDIF ENDIF ENDIF ENDFUNCTION

Tree Traversals

Three standard recursive traversals — each visits nodes in a different order.

TraversalOrderMnemonicOutput for example BST
In-orderLeft → Root → RightLNR20, 30, 40, 50, 60, 70, 80 (sorted!)
Pre-orderRoot → Left → RightNLR50, 30, 20, 40, 70, 60, 80
Post-orderLeft → Right → RootLRN20, 40, 30, 60, 80, 70, 50
Cambridge Pseudocode PROCEDURE InOrder(Current : INTEGER) IF Current <> 0 THEN InOrder(LeftPtr[Current]) OUTPUT NodeData[Current] InOrder(RightPtr[Current]) ENDIF ENDPROCEDURE PROCEDURE PreOrder(Current : INTEGER) IF Current <> 0 THEN OUTPUT NodeData[Current] PreOrder(LeftPtr[Current]) PreOrder(RightPtr[Current]) ENDIF ENDPROCEDURE PROCEDURE PostOrder(Current : INTEGER) IF Current <> 0 THEN PostOrder(LeftPtr[Current]) PostOrder(RightPtr[Current]) OUTPUT NodeData[Current] ENDIF ENDPROCEDURE
Traversal Trace — In-Order on the BST [50, 30, 70, 20, 40, 60, 80]
1
InOrder(50) → call InOrder(30) first
2
InOrder(30) → call InOrder(20) first
3
InOrder(20) → no left child → output 20 → no right child → return
4
Back at 30 → output 30 → call InOrder(40)
5
InOrder(40) → no children → output 40 → return
6
Back at 50 → output 50 → call InOrder(70)
7
InOrder(70) → InOrder(60) → output 60 → output 70 → InOrder(80) → output 80

Final output: 20, 30, 40, 50, 60, 70, 80 — in sorted order.

In-order always gives sorted output from a BST. This property is why BSTs are useful for maintaining sorted data with efficient O(log n) insertion and lookup (on a balanced tree).
Exam tip: When asked to trace a traversal, work through it methodically using a call stack. For in-order, think "go as far left as possible, then output, then go right." Cambridge often asks you to state the output of a traversal — practice on a diagram before the exam.

7 Hash Tables

A hash table stores key-value pairs and uses a hash function to compute an index (bucket) for each key. This enables O(1) average-case lookup, insertion, and deletion — much faster than searching a sorted array or BST.

2. Open Addressing (Linear Probing)

Hash Table — Pseudocode

Cambridge Pseudocode DECLARE Table : ARRAY[0:6] OF INTEGER DECLARE MaxSize : INTEGER MaxSize ← 7 // Initialise all slots to -1 (empty) FOR i ← 0 TO MaxSize - 1 Table[i] ← -1 NEXT i FUNCTION HashFunction(Key : INTEGER) RETURNS INTEGER RETURN Key MOD MaxSize ENDFUNCTION PROCEDURE Insert(Key : INTEGER) Index ← HashFunction(Key) // Linear probing: find next empty slot WHILE Table[Index] <> -1 DO Index ← (Index + 1) MOD MaxSize ENDWHILE Table[Index] ← Key ENDPROCEDURE FUNCTION Search(Key : INTEGER) RETURNS INTEGER // Returns index, or -1 if not found Index ← HashFunction(Key) Checked ← 0 WHILE Table[Index] <> Key AND Checked < MaxSize DO Index ← (Index + 1) MOD MaxSize Checked ← Checked + 1 ENDWHILE IF Table[Index] = Key THEN RETURN Index ELSE RETURN -1 ENDIF ENDFUNCTION

Properties of a Good Hash Function

  • Produces a valid index (within the table bounds)
  • Distributes keys evenly across the table — minimises clustering
  • Is fast to compute — O(1)
  • Is deterministic — the same key always produces the same index

Practice Question 5 — 4 marks

Compare a linked list and an array as implementations for a stack. State one advantage and one disadvantage of each.

Array-based stack:

  • Advantage: O(1) push and pop — no memory allocation per operation; fast due to cache locality of contiguous memory.
  • Disadvantage: Fixed maximum size must be declared in advance — stack overflow if the bound is exceeded; wastes memory if the stack is usually small.

Linked list-based stack:

  • Advantage: Dynamic size — can grow as large as available memory allows with no pre-declared limit.
  • Disadvantage: Each node requires extra memory for a pointer; heap allocation per push is slower than simple index increment; poorer cache performance.
Practice Question 6 — 3 marks

State the output of a pre-order and a post-order traversal of this tree:

        A
       / \
      B   C
     / \
    D   E

Pre-order (Root → Left → Right):
A, B, D, E, C

Post-order (Left → Right → Root):
D, E, B, C, A

Tip: Pre-order visits root before children (useful for copying a tree). Post-order visits root last (useful for deleting a tree — children deleted before parent).

ADT Quick Ref

  • Stack — LIFO, push/pop top
  • Queue — FIFO, enqueue rear / dequeue front
  • Linked List — dynamic, pointer-linked nodes
  • BST — left < root < right
  • Hash Table — key MOD size → O(1) avg

Tree Traversals

  • In-order: L → N → R (sorted)
  • Pre-order: N → L → R (copy)
  • Post-order: L → R → N (delete)

Primitive Types

  • INTEGER — whole number
  • REAL — decimal
  • CHAR — one character
  • STRING — text
  • BOOLEAN — TRUE/FALSE
  • DATE — calendar date

Common Mistakes

  • Forgetting TopOfStack starts at 0 (empty) not 1
  • Linear queue: not using circular logic — wastes slots
  • Mixing up in-order vs pre-order in traversals
  • Hash table: forgetting MOD wraps index into range
  • Arrays in pseudocode are 1-indexed — not 0
  • BST left < root ≤ right (some texts use strict <, know your convention)
