The four pillars of computational thinking, complexity classes (P, NP, NP-Complete), tractable and intractable problems, backtracking search, dynamic programming with memoisation, and heuristic approximation algorithms — how computer scientists approach hard problems systematically.

1 Computational Thinking

Computational thinking is a problem-solving approach that applies concepts from computer science to formulate problems in a way that can be solved systematically — by a computer or by a human following a process. It is not about coding; it is about thinking clearly about problems.

Computational thinking has four overlapping pillars:

Decomposition

Breaking a complex problem into smaller, more manageable sub-problems, each of which can be solved independently.

Example: Building a website = design UI + write back-end + set up database + write tests.

Abstraction

Identifying and focusing on the essential features of a problem while ignoring irrelevant detail. Creates a simplified model.

Example: A map abstracts the real world — it shows roads and landmarks but ignores grass, trees, and weather.

Pattern Recognition

Identifying similarities, trends, or regularities within or across problems. Recognised patterns can be reused rather than re-solved from scratch.

Example: Sorting algorithms all share the pattern of comparing and rearranging elements — once understood, this transfers to new contexts.

Algorithmic Thinking

Developing a step-by-step, unambiguous set of instructions — an algorithm — that solves the problem and can be implemented by any executor (computer or human).

Example: Writing pseudocode for binary search so that any programmer can implement it in any language.

Why Computational Thinking Matters

A problem that cannot be precisely specified cannot be solved by a computer. Computational thinking forces clarity: before writing any code, a programmer must decompose the problem, abstract away irrelevant detail, recognise similar solved problems, and design an algorithm. Poor thinking at this stage leads to incorrect or inefficient programs regardless of how well the code is written.

Worked Example — Applying the Four Pillars

Problem: Design a system to manage student exam registrations at a school.

1
Decomposition: Sub-problems — student database, subject/exam database, registration form, conflict detection (student can't sit two exams at the same time), timetable generation, reporting.
2
Abstraction: Represent each student as {ID, name, year group, subjects registered}. Ignore irrelevant data (home address, parent names) for this system.
3
Pattern recognition: Conflict detection resembles a graph colouring problem — a known problem in CS with established solutions.
4
Algorithmic thinking: Design procedures for: add_student(), register_subject(), check_conflicts(), generate_timetable().
Practice Question

A programmer is designing a navigation app that gives driving directions. Explain how each of the four pillars of computational thinking could be applied to this problem. (4 marks)

Decomposition: Split into sub-problems — map data storage, GPS position reading, route calculation, turn-by-turn instruction generation, real-time traffic updates, re-routing if driver deviates. (1 mark)

Abstraction: Represent the road network as a weighted graph (nodes = junctions, edges = roads, weights = distance/time) — ignoring irrelevant real-world details like road surfaces, building heights, or pavement widths. (1 mark)

Pattern recognition: Recognise that finding the fastest route is a shortest-path problem — a well-known problem for which algorithms (e.g., Dijkstra's) already exist. (1 mark)

Algorithmic thinking: Design a step-by-step procedure for each sub-problem, e.g., Dijkstra's algorithm for route calculation — an unambiguous set of steps that can be implemented in code. (1 mark)

2 Complexity & Tractability

Not all problems are equally hard. Computational complexity theory studies how the resources required to solve a problem (time, space) grow as the input size increases. This tells us which problems are feasibly solvable and which are not — even with very fast computers.

Big-O notation describes the worst-case growth rate of an algorithm's time requirement as input size n increases. It ignores constants and lower-order terms — only the dominant growth factor matters.

Scale matters: An O(2ⁿ) algorithm with n = 300 would require more operations than there are atoms in the observable universe — even running every computer on Earth for the age of the universe would not be enough. This is why complexity class matters for real-world feasibility.

Tractable and Intractable Problems

TermDefinitionGrowth RateExamples
Tractable Can be solved in polynomial time — O(nᵏ) for some fixed k. Practically solvable for realistic input sizes. O(1), O(log n), O(n), O(n²), O(n³), etc. Sorting, shortest path (Dijkstra), matrix multiplication
Intractable No polynomial-time algorithm is known. Grows faster than any polynomial — exponential or factorial. Practically unsolvable for large inputs. O(2ⁿ), O(n!), etc. Travelling Salesman Problem (exact), Boolean satisfiability, graph colouring
Key distinction: Tractable does not mean fast — O(n¹⁰⁰) is technically polynomial but practically useless. In practice, tractable usually means O(n²) or O(n³) at worst. The important dividing line is whether a polynomial-time algorithm exists at all.

Complexity Classes: P and NP

Class P

The set of all decision problems (yes/no questions) solvable in polynomial time by a deterministic algorithm. These are the tractable problems.

Examples: Is this list sorted? (O(n)), Does this graph have an Euler path? (O(E)), Is this number prime? (O(n³) via AKS).

Class NP

The set of all decision problems whose solutions can be verified in polynomial time, even if finding a solution may take exponential time.

Example: Given a proposed tour for TSP, you can verify its total cost in O(n) — but finding the optimal tour is not known to be doable in polynomial time.

NP-Complete

The hardest problems within NP. Every NP problem can be reduced to an NP-Complete problem in polynomial time. If any NP-Complete problem were solved in polynomial time, all NP problems would be too.

Examples: Boolean Satisfiability (SAT), Travelling Salesman, Graph Colouring, Knapsack (decision version).

P = NP? (Open Problem)

The most famous unsolved problem in computer science. If P = NP, every problem whose solution is easily verifiable could also be easily solved. Most researchers believe P ≠ NP, but it has never been proved.

A $1,000,000 Millennium Prize is offered for a solution.

Relationship Between Classes

All P problems are also in NP (a solution can trivially be verified by re-solving it). Whether there are NP problems outside P is the P vs NP question.

Practice Question

(a) Explain the difference between a tractable and an intractable problem. (2 marks)

(b) A student claims that if a faster computer were built, the Travelling Salesman Problem would become tractable. Evaluate this claim. (3 marks)

(a) A tractable problem has an algorithm that solves it in polynomial time — O(nᵏ) — so it remains feasible even for large inputs. (1 mark) An intractable problem has no known polynomial-time algorithm; the best known algorithms grow exponentially or factorially, making them impractical for large inputs regardless of hardware. (1 mark)

(b) The claim is incorrect (or at best misleading). A faster computer reduces the constant factor of execution but does not change the growth rate of the algorithm. (1 mark) For an exponential O(2ⁿ) algorithm, doubling CPU speed only allows solving a problem with n+1 more cities — if n=50 was previously the limit, n=51 would now be reachable. This negligible gain does not make the problem tractable. (1 mark) The TSP would only become tractable if a polynomial-time algorithm were discovered — a hardware improvement cannot change the mathematical complexity class of the problem. (1 mark)

3 Backtracking

Backtracking is a systematic algorithm design technique for finding solutions by exploring a tree of possibilities. It incrementally builds a solution candidate and abandons (backtracks from) any partial candidate as soon as it determines the candidate cannot possibly lead to a valid solution. This avoids the need to enumerate all possibilities exhaustively.

How Backtracking Works

  1. Choose the next decision point (e.g., assign a value to the next variable).
  2. Constrain — check whether the current partial assignment violates any constraint.
  3. If valid: recurse to the next decision point.
  4. If invalid (or no more options at this point): backtrack — undo the last choice and try the next option.
  5. If all variables are assigned with no constraint violation: a solution has been found.
Analogy: Backtracking is like solving a maze — you walk forward, and whenever you hit a dead end you retrace your steps to the last junction and try a different path.

Example: Graph Colouring

The graph colouring problem asks: can we assign colours to nodes of a graph such that no two adjacent nodes share the same colour, using at most k colours?

Consider a graph with 4 nodes (A, B, C, D) and edges A–B, A–C, B–C, B–D. Using 3 colours (R, G, B):

Assign A = R   Assign B = G (B adjacent to A=R, so not R)     Assign C = B (C adjacent to A=R and B=G, so must be B)       Assign D = R (D adjacent to B=G only, so R is valid)         ✓ Solution found: A=R, B=G, C=B, D=R

Example: Maze Solving

From a start cell, try moving in each direction (N, E, S, W). Mark the cell as visited. If a direction leads to a dead end or a visited cell, backtrack to the previous cell and try the next direction.

Start(1,1) → move East → (1,2) → move East → (1,3) → move South → (2,3) → move South → WALL — backtrack to (2,3) → move East → (2,4) → move South → EXIT found!

Backtracking Pseudocode (General Pattern)

PROCEDURE Backtrack(candidate)
  IF IsComplete(candidate) THEN
    Output(candidate) // solution found
    RETURN
  ENDIF
  FOR EACH choice IN GetChoices(candidate) DO
    IF IsValid(candidate, choice) THEN
      Apply(candidate, choice)
      Backtrack(candidate) // recurse
      Undo(candidate, choice) // undo if no solution found
    ENDIF
  ENDFOR
ENDPROCEDURE

Efficiency and Pruning

In the worst case, backtracking explores all possibilities — O(bᵈ) where b is the branching factor and d is the depth. However, good pruning (detecting invalid partial solutions early) can drastically reduce the actual search space:

  • Constraint propagation: When a value is assigned, immediately eliminate impossible values for other variables (reduces future branching).
  • Ordering: Try the most constrained variable first — fail fast rather than discovering a contradiction deep in the tree.

Applications: Sudoku solver, n-queens problem, graph colouring, maze solving, crossword puzzle solving, scheduling with hard constraints.

Practice Question

Describe how backtracking could be used to solve a Sudoku puzzle. Your answer should explain what constitutes a "valid" partial solution, what "backtracking" means in this context, and why backtracking is more efficient than brute force. (4 marks)

Process: Start with the given clues. For each empty cell, try each digit 1–9 in turn. After placing a digit, check whether it violates any constraint. (1 mark)

Valid partial solution: No digit appears more than once in its row, column, or 3×3 box. (1 mark)

Backtracking: If a digit leads to a violation (or all 9 digits for that cell have been tried without success), undo the last placement and try the next digit for that cell. If no digit works for the current cell, backtrack further to the previous cell. (1 mark)

Efficiency over brute force: A pure brute-force approach would generate all 9⁸¹ possible digit arrangements (≈ 2 × 10⁷⁷) and check each. Backtracking abandons invalid paths as soon as a constraint is violated — for a typical Sudoku puzzle, the vast majority of partial solutions are rejected early, making the actual search space tiny compared to brute force. (1 mark)

4 Dynamic Programming

Dynamic programming (DP) is an algorithm design technique for solving optimisation problems that have two key properties:

  • Overlapping subproblems: The problem can be broken into sub-problems which recur multiple times. Naïve recursion would recompute the same sub-problems repeatedly.
  • Optimal substructure: The optimal solution to the problem can be constructed from the optimal solutions to its sub-problems.

DP solves each sub-problem exactly once and stores the result, avoiding redundant recomputation. This transforms exponential-time recursive solutions into polynomial-time ones.

Two Approaches

ApproachNameMethodWhen to use
Top-down Memoisation Write a recursive solution; cache results in a table. Before computing a sub-problem, check if it's already been computed and return the cached value. When only some sub-problems will actually be needed; natural recursive structure
Bottom-up Tabulation Fill a table iteratively, starting from the smallest sub-problems and working up to the full problem. No recursion — each cell depends only on previously computed cells. When all sub-problems must be solved; avoids recursion overhead

Classic Example: Fibonacci Sequence

The naive recursive approach to computing Fibonacci numbers has O(2ⁿ) time complexity — it recomputes the same values exponentially many times. Dynamic programming reduces this to O(n).

Worked Example — Fibonacci with Memoisation

Naive recursion (slow): fib(5) calls fib(4) and fib(3). fib(4) calls fib(3) and fib(2). fib(3) is computed twice — and this redundancy compounds at every level.

FUNCTION fib(n)
  IF memo[n] <> -1 THEN RETURN memo[n] // cached?
  IF n <= 1 THEN RETURN n // base case
  result ← fib(n-1) + fib(n-2)
  memo[n] ← result // store result
  RETURN result
ENDFUNCTION

With memoisation, each fib(k) is computed exactly once. Total calls: O(n) instead of O(2ⁿ).

Worked Example — Fibonacci with Tabulation (Bottom-up)

Compute fib(0) to fib(7) iteratively:

n01234567
fib(n) 0112 35813

Each cell = sum of the two cells to its left. Fill left to right — O(n) time, O(n) space (reducible to O(1) if only the last two values are stored).

Classic Example: 0/1 Knapsack Problem

Given n items each with a weight wᵢ and value vᵢ, and a knapsack with capacity W, find the maximum value that can be packed without exceeding W. Each item can be included (1) or excluded (0) — not split.

Recurrence:

dp[i][w] = max value using items 1..i with capacity w

dp[i][w] = dp[i-1][w] if wᵢ > w (item too heavy — skip)
dp[i][w] = max(dp[i-1][w], vᵢ + dp[i-1][w-wᵢ]) if wᵢ ≤ w (skip or take item)
Worked Example — Knapsack DP Table

Items: A (weight=2, value=3), B (weight=3, value=4), C (weight=4, value=5). Capacity W = 5.

Item \ Capacity012345
None000000
A (w=2,v=3)003333
B (w=3,v=4)003447
C (w=4,v=5)003457

Answer: dp[3][5] = 7 — take item A (value 3) and item B (value 4), total weight = 5, total value = 7.

✓
At capacity 5 with item B available: take B (4) + best with remaining capacity 2 = dp[A][2] = 3 → total 7. Skip C (weight 4, only 1 capacity left when B is taken).
Exam tip: For Cambridge Paper 3, you should be able to: (1) explain what overlapping subproblems and optimal substructure mean; (2) distinguish memoisation from tabulation; (3) trace a simple DP table for Fibonacci or a small knapsack — you will not be asked to derive complex recurrences from scratch, but you must be able to fill in a given table and read off the answer.
Practice Question

Explain why computing Fibonacci numbers using simple recursion is inefficient and how dynamic programming (memoisation) solves this problem. Use fib(6) as an example. (4 marks)

Inefficiency: Simple recursion recomputes the same sub-problems multiple times. For fib(6), fib(4) is called twice, fib(3) is called three times, fib(2) is called five times, etc. The number of calls grows exponentially — O(2ⁿ) — because the recursion tree branches twice at each level with no memory of previous results. (2 marks)

Memoisation fix: Before computing fib(n), check a memo table (initially all -1 / empty). If the result has already been calculated, return it immediately. Otherwise compute it, store the result in the table, then return it. (1 mark) Each value fib(0) through fib(6) is now computed exactly once — reducing the total number of operations from O(2ⁿ) to O(n). For fib(6) this means 7 computations instead of 25. (1 mark)

5 Heuristics

For intractable problems, exact algorithms that guarantee optimal solutions are impractical for large inputs. A heuristic is an approximate method — a "rule of thumb" — that finds a good solution quickly, without guaranteeing that it is the best possible solution. The trade-off is: speed vs optimality.

Why Use Heuristics?

Exact AlgorithmHeuristic
Guarantees optimal (best possible) solutionFinds a good solution — may not be optimal
Exponential or factorial time — impractical for large nPolynomial time — fast even for large n
Required when the cost of a suboptimal solution is unacceptableAcceptable when a "good enough" solution is sufficient
Example: enumerate all routes for TSPExample: nearest-neighbour heuristic for TSP

The Travelling Salesman Problem (TSP)

A travelling salesman must visit n cities exactly once each and return to the starting city. The goal is to find the shortest possible tour. TSP is NP-Complete — the only exact algorithms run in O(n!) or O(2ⁿ · n) time. For n = 20 cities, brute force requires checking over 10¹⁷ routes.

Nearest-Neighbour Heuristic

The nearest-neighbour heuristic is a greedy approach to TSP:

  1. Start at any city (usually the first, or a specified starting city).
  2. At each step, move to the nearest unvisited city.
  3. After all cities have been visited, return to the starting city.
Worked Example — Nearest-Neighbour Heuristic

Cities: A, B, C, D, E. Distances (symmetric — same in both directions):

ABCDE
A—1081512
B10—6914
C86—117
D15911—5
E121475—

Starting at A:

1
At A: nearest unvisited = C (distance 8). Move to C. Visited: {A, C}.
2
At C: nearest unvisited = B (distance 6) or E (7). B is closer. Move to B. Visited: {A, C, B}.
3
At B: nearest unvisited = D (distance 9) or E (14). Move to D. Visited: {A, C, B, D}.
4
At D: only unvisited = E (distance 5). Move to E. Visited: {A, C, B, D, E}.
5
Return to A from E: distance 12.
A→8 C→6 B→9 D→5 E→12 A Total: 40

This heuristic tour has cost 40. The optimal tour may be shorter (e.g., A→C→E→D→B→A = 8+7+5+9+10 = 39), but finding it requires an exact algorithm. The heuristic gives a solution that is close to optimal in O(n²) time instead of O(n!).

Limitations of the Nearest-Neighbour Heuristic

  • Does not guarantee the optimal tour — the final "return" edge is not chosen greedily and can be very long.
  • The quality of the result depends on the starting city — different starts give different tours with different costs.
  • Can produce tours that are significantly worse than optimal (in theory, up to O(log n) times longer for adversarial inputs).

Other Heuristic Approaches

Greedy Algorithms

At each step, make the locally optimal choice. Fast but may miss the global optimum. Nearest-neighbour is a greedy heuristic for TSP. Works well when locally optimal choices lead to globally good solutions.

Simulated Annealing

Inspired by metallurgical annealing. Accepts worse solutions with a probability that decreases over time ("temperature"). This allows escape from local optima — produces better solutions than pure greedy at the cost of more computation.

Genetic Algorithms

Maintain a population of candidate solutions. Evolve them using selection (keep best), crossover (combine two solutions), and mutation (random change). Mimics biological evolution. Often finds very good solutions for combinatorial problems.

A* Search

Uses a heuristic function h(n) to estimate the cost from node n to the goal. Explores the most promising paths first. Guarantees an optimal solution if h(n) never overestimates the true cost (admissible heuristic). Used in pathfinding.

Exam tip: The Cambridge syllabus specifically names the nearest-neighbour heuristic for TSP. Be ready to: (1) apply it step-by-step to a given distance matrix; (2) state that it does not guarantee an optimal solution; (3) explain why heuristics are used — because TSP is intractable and an exact algorithm would take too long for large inputs.

A delivery driver must visit 4 depots (P, Q, R, S) starting and ending at P. The distances in kilometres are:

PQRS
P—593
Q5—47

(a) Apply the nearest-neighbour heuristic starting at P. Show each step and state the total distance. (3 marks)

Start at P. Nearest unvisited: S (distance 3). Move to S. (1 mark)

Question 1 — 4 marks

Explain the four pillars of computational thinking and give one example of each applied to the problem of designing a search engine. (4 marks)

Decomposition: Break the search engine into sub-problems — web crawler (fetching pages), indexer (storing word-to-page mappings), ranking algorithm (ordering results), query parser (understanding user input), UI. (1 mark)

Abstraction: Represent each web page as a document with a URL, a set of words, and a set of outbound links — ignoring images, CSS, or JavaScript code unless relevant. (1 mark)

Pattern recognition: Recognise that ranking pages by importance resembles the problem of finding influential nodes in a graph — a known problem solved by algorithms like PageRank. (1 mark)

Algorithmic thinking: Design step-by-step procedures: e.g., BFS crawler algorithm, inverted index construction, PageRank iteration — unambiguous enough to be implemented. (1 mark)

Question 2 — 5 marks

(a) Explain what is meant by NP-Complete. (2 marks)

(b) The Boolean Satisfiability Problem (SAT) is NP-Complete. A programmer argues that because modern computers are fast, SAT can be solved quickly for any input. Evaluate this claim. (3 marks)

(a) NP-Complete problems are the hardest problems in the class NP — problems whose solutions can be verified in polynomial time. (1 mark) A problem is NP-Complete if every other problem in NP can be reduced to it in polynomial time. If any NP-Complete problem could be solved in polynomial time, all NP problems could be. (1 mark)

(b) The claim is incorrect. The best known algorithms for SAT have worst-case exponential time complexity — O(2ⁿ) in the number of variables. (1 mark) A faster computer reduces the constant factor but not the growth rate. For example, if a computer can check 10¹² assignments per second, solving an instance with n = 100 variables would require checking up to 2¹⁰⁰ ≈ 10³⁰ assignments — taking approximately 10¹⁸ seconds (far longer than the age of the universe). (1 mark) Modern SAT solvers use intelligent backtracking and heuristics that work well in practice on many real-world instances, but there exist inputs for which they still require exponential time — the worst case has not been eliminated. (1 mark)

Question 3 — 4 marks

Trace a backtracking algorithm to colour the following graph using at most 2 colours (Red and Blue) such that no two adjacent nodes share the same colour. Nodes: W, X, Y, Z. Edges: W–X, X–Y, Y–Z.

Show each assignment and any backtracking steps. State whether a valid 2-colouring exists. (4 marks)

Assign W = Red. (1 mark)

X is adjacent to W (Red) → assign X = Blue. (1 mark)

Y is adjacent to X (Blue) → assign Y = Red. (1 mark)

Z is adjacent to Y (Red) → assign Z = Blue. No violations — solution found: W=Red, X=Blue, Y=Red, Z=Blue. (1 mark)

No backtracking was needed here because this graph is a path (bipartite), which is always 2-colourable. A valid 2-colouring exists.

If the graph had a triangle (W–X, X–Y, W–Y), 2-colouring would be impossible and backtracking would exhaust all options before reporting "no solution."

Question 5 — 4 marks

Explain the difference between dynamic programming and backtracking as problem-solving strategies. Include in your answer the type of problem each is suited to, and the key technique that makes dynamic programming more efficient than naive recursion. (4 marks)

Backtracking is suited to constraint satisfaction problems where you must find a valid assignment of values to variables (e.g., graph colouring, Sudoku). It works by building a solution incrementally and abandoning branches as soon as a constraint is violated. It does not store intermediate results. (1 mark)

Dynamic programming is suited to optimisation problems with overlapping subproblems and optimal substructure (e.g., shortest path, knapsack, Fibonacci). It decomposes the problem into subproblems and solves each optimally. (1 mark)

Key technique — memoisation/tabulation: DP stores the result of each subproblem in a table the first time it is computed. Subsequent requests for the same subproblem return the cached value instantly rather than recomputing it. This eliminates the redundant recomputation that makes naive recursion exponentially slow. (2 marks)

Question 6 — 3 marks

State what is meant by a heuristic algorithm, and explain the trade-off involved in using a heuristic instead of an exact algorithm. Give one example. (3 marks)

Definition: A heuristic is an approximate algorithm that finds a good solution to a problem without guaranteeing that the solution is optimal. It applies a "rule of thumb" to make decisions quickly. (1 mark)

Trade-off: A heuristic sacrifices optimality (the solution may not be the best possible) in exchange for speed — it runs in polynomial time whereas the exact algorithm may require exponential or factorial time. For large intractable problems where an exact solution is practically impossible, a near-optimal solution found quickly is far more useful than waiting for an exact answer that never arrives. (1 mark)

Example: The nearest-neighbour heuristic for the Travelling Salesman Problem — at each step move to the nearest unvisited city. This runs in O(n²) time but may not produce the shortest possible tour, unlike the brute-force exact method which runs in O(n!). (1 mark)

4 Pillars

  • Decomposition — break into sub-problems
  • Abstraction — ignore irrelevant detail
  • Pattern recog. — reuse known solutions
  • Algorithmic — step-by-step procedure

Complexity Classes

  • P — solvable in poly time (tractable)
  • NP — verifiable in poly time
  • NP-Complete — hardest in NP
  • Intractable — no poly-time algorithm known

DP Quick Ref

  • Needs: overlapping subproblems + optimal substructure
  • Memoisation — top-down, cache recursion
  • Tabulation — bottom-up, fill table iteratively

Big-O Growth

  • O(1) < O(log n) < O(n)
  • O(n log n) < O(n²) < O(2ⁿ)
  • O(n!) — worst / intractable

Common Mistakes

  • Intractable ≠ impossible — it means no polynomial-time algorithm is known
  • Faster hardware doesn't change complexity class — only an algorithmic breakthrough can
  • Backtracking is NOT brute force — it prunes invalid branches early
  • Nearest-neighbour does NOT guarantee the shortest tour
  • Memoisation = top-down + cache; Tabulation = bottom-up + table
