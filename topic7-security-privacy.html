Security, Privacy & Data Integrity

Encryption, digital certificates, SSL/TLS, common threats, firewalls, hashing, and data validation techniques for keeping systems and data secure.

1 Encryption

Encryption converts plaintext into ciphertext using an algorithm and a key, so that only authorised parties can read the original data. It protects data in transit (e.g. over the Internet) and at rest (e.g. on disk).

Symmetric Encryption

Both sender and receiver use the same key to encrypt and decrypt data.

  • Example algorithm: AES (Advanced Encryption Standard)

  • Disadvantage: Key distribution problem — how do you securely share the key with the other party?

Asymmetric Encryption

Uses a mathematically linked key pair: a public key (shared openly) and a private key (kept secret). Data encrypted with the public key can only be decrypted with the private key.

How HTTPS combines both: Asymmetric encryption is used during the TLS handshake to securely exchange a symmetric session key. The rest of the session then uses that faster symmetric key.

Digital Signatures

A digital signature proves that a message came from a specific sender and has not been altered. It uses asymmetric encryption in reverse: the sender encrypts a hash of the message with their private key.

How a Digital Signature Works
1
Sender creates a hash (digest) of the message using a hashing algorithm (e.g. SHA-256).
2
Sender encrypts the hash with their private key → this is the digital signature.
3
Message + signature are sent to the receiver.
4
Receiver decrypts the signature using the sender's public key to recover the original hash.
5
Receiver independently hashes the received message and compares it to the decrypted hash. If they match → message is authentic and unaltered.
  • Authenticity: Only the sender with the private key could have created the signature
  • Integrity: Any change to the message would produce a different hash — mismatch detected
  • Non-repudiation: Sender cannot deny sending the message
Exam tip: Be precise about which key does what. For confidentiality: encrypt with recipient's public key, decrypt with recipient's private key. For a digital signature: sign with sender's private key, verify with sender's public key.

2 Digital Certificates & SSL/TLS

Digital Certificates

A digital certificate is an electronic document that binds a public key to an identity (e.g. a website). It is issued and digitally signed by a trusted Certificate Authority (CA).

A certificate typically contains:

  • The subject's name (e.g. domain name)
  • The subject's public key
  • The CA's identity and digital signature
  • Validity period (expiry date)
  • Serial number and version
Why certificates matter: Without certificates, an attacker could present their own public key and pretend to be a trusted website (man-in-the-middle attack). The CA's signature on the certificate proves the public key really belongs to the claimed identity.

SSL / TLS

TLS (Transport Layer Security) — and its predecessor SSL — provides encrypted, authenticated communication over a network. HTTPS uses TLS to secure web traffic.

The TLS handshake (simplified):

TLS Handshake
1
Client Hello: Browser sends supported TLS version and cipher suites.
2
Server Hello: Server responds with chosen cipher suite and sends its digital certificate.
3
Certificate verification: Browser verifies the certificate against trusted CAs. If valid, it trusts the server's public key.
4
Key exchange: Browser uses the server's public key (asymmetric) to securely send a pre-master secret.
5
Session keys generated: Both sides derive the same symmetric session key from the pre-master secret.
6
Secure communication begins: All further data is encrypted with the symmetric session key.
Exam tip: Know that TLS uses asymmetric encryption to exchange the symmetric session key, then switches to faster symmetric encryption for actual data. You may be asked to explain why both types are used.

3 Threats to Systems & Data

Understanding the types of threats helps in selecting appropriate countermeasures.

Malware

Virus

Attaches itself to legitimate files. Spreads when infected file is executed. Can corrupt data or system files.

Worm

Self-replicates across networks without user action. Consumes bandwidth. Does not need a host file.

Trojan Horse

Disguised as legitimate software. When run, performs malicious actions (e.g. creates backdoor, steals data).

Ransomware

Encrypts victim's files and demands payment for the decryption key. Can destroy businesses.

Spyware

Secretly monitors user activity and collects sensitive data (keystrokes, passwords, browsing history).

Rootkit

Hides deep in the OS, giving attackers persistent privileged access while concealing their presence.

Social Engineering

Social engineering exploits human psychology rather than technical vulnerabilities to gain unauthorised access.

AttackDescription
PhishingFake emails / websites trick users into revealing credentials or downloading malware. Often impersonates trusted organisations.
Spear phishingTargeted phishing aimed at a specific individual using personalised information to appear credible.
VishingVoice phishing — attackers call victims posing as bank or IT support to extract information.
PretextingCreating a fabricated scenario (pretext) to gain a victim's trust and extract information.
BaitingLeaving infected USB drives in public; curiosity leads victims to plug them in.

Technical Attacks

AttackDescriptionCountermeasure
SQL InjectionMalicious SQL code inserted into input fields to manipulate the database (e.g. bypass login, dump data).Parameterised queries, input validation
DDoSDistributed Denial of Service — overwhelming a server with traffic from many compromised machines (botnet).Rate limiting, traffic filtering, CDN
Man-in-the-MiddleAttacker secretly intercepts and potentially alters communications between two parties.TLS/HTTPS, certificate pinning

Key distinction: A virus needs a host file to spread; a worm is self-contained and spreads on its own. A Trojan does not self-replicate — it relies on the user running it.

4 Network Security Measures

🔥

Firewall

Hardware or software that monitors and filters network traffic using rules based on IP addresses, port numbers, and protocols. Blocks unauthorised connections.

🔀

Proxy Server

Acts as an intermediary between clients and the Internet. Can filter content, cache web pages, hide internal IP addresses, and provide anonymity.

🔐

VPN

Virtual Private Network creates an encrypted tunnel over a public network. Extends a private network securely, hiding traffic from eavesdroppers.

🔑

Multi-Factor Authentication

Requires two or more verification factors: something you know (password), have (phone/token), or are (biometrics). Greatly reduces account compromise risk.

🛡️

Intrusion Detection System

IDS monitors network traffic for suspicious patterns and alerts administrators. An IPS (prevention system) can also block detected threats automatically.

📡

Access Control Lists

Rules that specify which users or systems are permitted to access resources. Implemented at network level (routers/switches) or OS level (file permissions).

Firewall Rules — How They Work

A firewall inspects each packet against an ordered list of rules:

  • Packet filtering: Checks source/destination IP, port, and protocol. Stateless — does not track connection state.
  • Stateful inspection: Tracks the state of active connections. Only allows packets that are part of an established, legitimate session.
  • Application layer (proxy firewall): Understands specific application protocols (HTTP, FTP) and can filter based on content.
Firewall vs Proxy: A firewall controls which traffic is allowed; a proxy server relays requests on behalf of clients. A firewall works at the network perimeter; a proxy also provides caching and anonymity.

Physical Security

Technical measures alone are insufficient. Physical security controls protect hardware from theft or tampering:

  • Locked server rooms with badge/biometric access
  • CCTV surveillance
  • Cable locks for workstations
  • Secure disposal of storage media (degaussing, physical destruction)

5 Data Validation & Verification

These two concepts are often confused. They serve different purposes:

ValidationVerification
PurposeChecks that data is sensible, reasonable, and within expected rulesChecks that data has been entered correctly — matches the source
Who checksThe computer / software automaticallyA human or a second entry compared by the system
ExampleAge must be between 0 and 120Enter password twice; system checks both match
LimitationCannot check if data is true — a valid age of 25 could still be wrong if the person is actually 30Cannot check if the original data is correct — only that it was copied consistently

Types of Validation Check

CheckWhat it testsExample
Range checkValue falls within an acceptable minimum and maximumExam score must be 0–100; month must be 1–12
Type checkData is the correct data typePhone number must contain only digits; name must be a string
Presence checkA required field is not left emptyUsername field cannot be blank on a registration form
Format checkData matches an expected patternEmail must match pattern user@domain.ext; date must be DD/MM/YYYY
Length checkData has the correct number of charactersPassword must be 8–20 characters; UK postcode has 6–7 characters
Check digitCalculated digit appended to data to detect input errorsLast digit of ISBN-13, barcode, credit card number (Luhn algorithm)
Lookup checkValue exists in a predefined list of acceptable valuesCountry code must exist in a list of valid country codes
Check Digit Example — Simple Modulo 11

To verify the number 1234567 with check digit:

1
Multiply digits by weights 7, 6, 5, 4, 3, 2: (1×7)+(2×6)+(3×5)+(4×4)+(5×3)+(6×2) = 7+12+15+16+15+12 = 77
2
77 mod 11 = 0. Check digit = 0 (if remainder = 0) or 11 − remainder otherwise.
3
Receiver performs the same calculation; if the result matches the appended check digit, no errors detected.

Check digits detect single transposition errors (e.g. 123 entered as 132) but not all errors.

Verification Methods

  • Double entry: User types the data twice (e.g. email address, password). System checks both entries match before accepting.
  • Proofreading: A human reads the entered data against the original source document to spot discrepancies.
Exam tip: Never say validation "checks data is correct." It only checks data is reasonable and within rules. A valid-looking age of 150 would fail a range check, but a valid age of 45 that is actually wrong would pass — validation cannot detect that.

6 Hashing

A hash function maps data of any size to a fixed-length output called a hash, digest, or fingerprint.

Properties of a Good Hash Function

  • Deterministic: The same input always produces the same hash.
  • One-way (pre-image resistant): Computationally infeasible to reverse a hash to find the original input.
  • Avalanche effect: A small change in the input produces a completely different hash.
  • Collision resistant: Computationally infeasible to find two different inputs that produce the same hash.
  • Fixed-length output: SHA-256 always outputs 256 bits regardless of input size.

Password Storage

Passwords should never be stored in plaintext. Instead:

Secure Password Storage with Salt
1
User creates password "mypassword".
2
System generates a random salt (e.g. "x7Qp") — unique per user.
3
System computes: hash("mypassword" + "x7Qp") → stores the hash AND the salt.
4
On login: system retrieves the salt, hashes entered password + salt, compares to stored hash.

Why salt? Without salting, identical passwords produce identical hashes — attackers can use pre-computed lookup tables (rainbow tables) to reverse common hashes. A unique salt per user defeats rainbow table attacks.

Uses of Hashing

UseHow Hashing Helps
Password storageStore hash (+ salt), not plaintext. Even if the database is stolen, passwords are not exposed.
Data integrityHash a file before and after transmission; compare hashes to detect any corruption or tampering.
Digital signaturesHash the message first (to get a short fixed-length value), then sign the hash with a private key.
BlockchainEach block contains the hash of the previous block, creating a tamper-evident chain.
Hashing ≠ Encryption: Encryption is reversible (with the correct key). Hashing is a one-way function — you cannot decrypt a hash to get the original data. Never confuse the two in an exam answer.

📝 Exam Practice

Practice Question 1 — 4 marks

Explain the difference between symmetric and asymmetric encryption. State one advantage of each.

Symmetric: Both sender and receiver use the same (shared) key to encrypt and decrypt data. Advantage: much faster than asymmetric — suitable for encrypting large amounts of data.

Asymmetric: Uses a key pair — a public key (freely shared) to encrypt and a private key (kept secret) to decrypt. Advantage: Solves the key distribution problem — public keys can be shared openly without compromising security.

Practice Question 2 — 5 marks

Describe how a digital signature is created and verified. Your answer should state what is produced at each stage.

Creating: (1) A hash of the message is computed using a hashing algorithm. (2) The hash is encrypted with the sender's private key — this encrypted hash is the digital signature. (3) The message and the digital signature are sent together.

Verifying: (4) The receiver decrypts the digital signature using the sender's public key to recover the hash. (5) The receiver independently hashes the received message. (6) If the two hashes match, the message is authentic (really from the sender) and has not been altered.

Practice Question 3 — 3 marks

State three different types of validation check that could be applied when a user enters a date of birth on a web form. For each, give an example of what it would check.

Any three from:

  • Format check — date must match the pattern DD/MM/YYYY
  • Range check — year must be between 1900 and the current year
  • Type check — entry must contain only digits and separators (not letters)
  • Presence check — the field cannot be left empty
  • Range check on month — month value must be 1–12
  • Range check on day — day value must be 1–31 (or appropriate for the month)
Practice Question 4 — 4 marks

Explain why passwords should be stored as salted hashes rather than plaintext or unsalted hashes.

Plaintext: If the database is breached, all passwords are immediately exposed.

Unsalted hashes: An attacker can use pre-computed rainbow tables to look up common hashes and reverse them to find passwords. Users with the same password produce the same hash, making patterns visible.

Salted hashes: A unique random salt is added to each password before hashing. This means even identical passwords produce different hashes. Rainbow tables become useless because they would need to be rebuilt for every possible salt, which is computationally infeasible.

Practice Question 5 — 2 marks

State the difference between validation and verification, giving one example of each.

Validation checks that data is reasonable and meets defined rules (done automatically by the computer). Example: checking that an age value is between 0 and 120.

Verification checks that data has been entered correctly, matching the original source. Example: a user types their email address twice and the system checks both entries are identical.

Key Formulae

  • Sign = encrypt(hash, private key)
  • Verify = decrypt(sig, public key)
  • Store: hash(password + salt)

Encryption Summary

  • Symmetric: 1 shared key, fast, AES
  • Asymmetric: key pair, slow, RSA
  • HTTPS: asymmetric to share session key → symmetric for data

Common Mistakes

  • Saying validation checks data is "correct" — it only checks data is reasonable
  • Confusing which key signs vs encrypts in digital signatures
  • Saying hashing is the same as encryption — hashing is one-way
  • Forgetting that a virus needs a host file but a worm does not
  • Saying TLS only uses asymmetric encryption — it uses both
