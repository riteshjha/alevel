Databases

Relational database concepts, entity-relationship modelling, normalisation to 3NF, SQL queries, DBMS features, and ACID transaction properties.

1 Database Concepts

A relational database organises data into tables (relations) of rows and columns, where relationships between tables are expressed through shared key values — not through physical pointers or file links.

Key Terminology

TermDefinitionExample
EntityA real-world object or concept about which data is storedStudent, Course, Lecturer
AttributeA property or characteristic of an entityStudentID, FirstName, DateOfBirth
Relation / TableA two-dimensional structure of rows and columns representing an entityThe Students table
Tuple / Row / RecordA single instance of an entity — one row in a tableA single student's data
Attribute / Column / FieldA single type of data stored across all recordsThe Grade column
Primary Key (PK)An attribute (or combination) that uniquely identifies each row. Must be unique and NOT NULL.StudentID
Foreign Key (FK)An attribute in one table that references the primary key in another table, creating a relationshipCourseID in Enrolments references CourseID in Courses
Composite KeyA primary key made up of two or more attributes that together are unique(StudentID, CourseID) in an Enrolments table
Referential IntegrityA foreign key value must either match an existing PK in the referenced table or be NULL — no orphan recordsCannot add an enrolment for a StudentID that does not exist

Why Relational Databases?

  • Reduce data redundancy: Each piece of information is stored once; other tables reference it via keys
  • Maintain data integrity: Constraints (PK, FK, NOT NULL) prevent invalid data entering the system
  • Data independence: The physical storage can change without affecting how applications query the data
  • Powerful querying: SQL can combine and filter data across multiple tables in complex ways
  • Concurrent access: Multiple users can read and write safely using transaction management
Flat file vs relational: A flat file stores everything in one table — data is repeated for every record (e.g. the course name appears on every student row for that course). A relational database stores each fact once, eliminating update anomalies.

An Entity-Relationship (ER) diagram is a visual model of the entities in a system and the relationships between them. It is produced during database design, before writing any SQL.

Resolving Many-to-Many Relationships

A M:N relationship cannot be directly implemented in a relational database. It must be decomposed into two 1:M relationships via a link table (also called a junction or associative table).

Example: Student ↔ Course
1
Identify the M:N: a Student can enrol on many Courses; a Course can have many Students.
2
Create a link table Enrolment with foreign keys to both Student and Course.
3
The composite key (StudentID, CourseID) uniquely identifies each enrolment, preventing duplicates.
4
Extra attributes of the relationship (e.g. Grade, EnrolDate) belong in the link table, not in either parent table.
Exam tip: In ER diagrams always label the relationship verb (e.g. "enrols on", "teaches") and both ends of the cardinality. Cambridge questions often ask you to add a link table — remember to include both FKs and any relationship attributes.

3 Normalisation

Normalisation is the process of organising a database to minimise data redundancy and avoid update anomalies. It proceeds through a series of normal forms, each building on the last.

Why Normalise?

An unnormalised table suffers from update anomalies:

  • Insertion anomaly: Cannot add data without also adding unrelated data (e.g. cannot add a new course unless a student enrols on it)
  • Deletion anomaly: Deleting one record unintentionally destroys other information (e.g. removing the last student on a course deletes the course details)
  • Update anomaly: Changing one fact requires updating many rows — and inconsistency arises if some rows are missed

Unnormalised Form (UNF)

Data exactly as it arrives — may contain repeating groups and no clear primary key.

OrderIDCustomerNameCustomerEmailItemsQuantities
1001Alice Smithalice@email.comPen, Notebook3, 1
1002Bob Jonesbob@email.comRuler2

Problem: the Items and Quantities columns hold multiple values — a repeating group.

First Normal Form (1NF)

A relation is in 1NF if:

  • Each cell contains a single atomic value (no repeating groups, no multi-valued attributes)
  • All values in a column are of the same type
  • Each row is unique (there is a primary key)
✗ Not in 1NF
OrderIDItemsQty
1001Pen, Notebook3, 1
✓ In 1NF
OrderIDItemQty
1001Pen3
1001Notebook1

Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and every non-key attribute is fully functionally dependent on the whole primary key (no partial dependencies).

Partial dependencies only arise when the primary key is composite. A non-key attribute that depends on only part of the composite key must be moved to a separate table.

Removing a Partial Dependency

Table: OrderItem(OrderID, ItemID, ItemName, Quantity)

Composite PK = (OrderID, ItemID)

Problem: ItemName depends only on ItemID — not on the full key (OrderID, ItemID). This is a partial dependency.

1
Remove ItemName from OrderItem.
2
Create a new table: Item(ItemID, ItemName).
3
Result in 2NF: OrderItem(OrderID, ItemID, Quantity) and Item(ItemID, ItemName).

Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and there are no transitive dependencies — every non-key attribute depends only on the primary key, not on another non-key attribute.

Removing a Transitive Dependency

Table: Employee(EmployeeID, Name, DepartmentID, DepartmentName)

Problem: DepartmentName depends on DepartmentID, which is not the PK — a transitive dependency.

1
Remove DepartmentName from Employee.
2
Create a new table: Department(DepartmentID, DepartmentName).
3
Result in 3NF: Employee(EmployeeID, Name, DepartmentID) and Department(DepartmentID, DepartmentName).

Normal Forms — Summary

Normal FormRequirementRemoves
1NFAtomic values; no repeating groups; has a PKMulti-valued / repeating attributes
2NFIn 1NF + no partial dependencies on composite PKPartial dependencies
3NFIn 2NF + no transitive dependenciesTransitive dependencies
Exam tip: To spot a transitive dependency, look for a non-key attribute that depends on another non-key attribute — not on the PK. Write the functional dependencies explicitly: "DepartmentName → DepartmentID → not the PK" signals a transitive dependency to remove.

4 Structured Query Language (SQL)

SQL is the standard language for querying and manipulating relational databases. Cambridge 9618 expects you to write and interpret SQL for SELECT, INSERT, UPDATE, DELETE, and JOIN operations.

SELECT — Retrieving Data

SQL -- Basic SELECT SELECT FirstName, LastName, Grade FROM Students WHERE Grade >= 70 ORDER BY Grade DESC; -- Select all columns SELECT * FROM Students; -- Eliminate duplicate rows SELECT DISTINCT Department FROM Employees; -- Pattern matching with LIKE SELECT * FROM Students WHERE LastName LIKE 'S%'; -- starts with S SELECT * FROM Students WHERE Email LIKE '%@gmail.com';

WHERE Clause Operators

OperatorMeaningExample
=, <>, <, >, <=, >=Comparison operatorsWHERE Age >= 18
AND, OR, NOTLogical operatorsWHERE Grade > 50 AND Attendance >= 80
BETWEEN … AND …Inclusive rangeWHERE Grade BETWEEN 60 AND 80
LIKEPattern match (% = any chars, _ = one char)WHERE Name LIKE 'A_i%'
IN (…)Matches any value in a listWHERE Status IN ('Active', 'Pending')
IS NULL / IS NOT NULLTests for missing valuesWHERE PhoneNumber IS NOT NULL

Aggregate Functions & GROUP BY

SQL -- Aggregate functions SELECT COUNT(*) AS TotalStudents FROM Students; SELECT AVG(Grade) AS AverageGrade FROM Students; SELECT MAX(Grade), MIN(Grade) FROM Students; SELECT SUM(Credits) FROM Courses; -- GROUP BY: one result row per group SELECT Department, COUNT(*) AS StaffCount FROM Employees GROUP BY Department; -- HAVING: filters groups (like WHERE but for aggregates) SELECT Department, AVG(Salary) AS AvgSalary FROM Employees GROUP BY Department HAVING AVG(Salary) > 40000;

JOIN — Combining Tables

A JOIN combines rows from two or more tables based on a related column.

SQL -- INNER JOIN: only rows with a match in both tables SELECT s.FirstName, s.LastName, c.CourseName, e.Grade FROM Students s INNER JOIN Enrolments e ON s.StudentID = e.StudentID INNER JOIN Courses c ON e.CourseID = c.CourseID WHERE e.Grade >= 60 ORDER BY s.LastName; -- LEFT JOIN: all rows from left table, NULLs where no match on right SELECT s.FirstName, e.CourseID FROM Students s LEFT JOIN Enrolments e ON s.StudentID = e.StudentID; -- Students with no enrolments appear with NULL CourseID

INSERT, UPDATE, DELETE

SQL -- INSERT a new row INSERT INTO Students (FirstName, LastName, Grade) VALUES ('Alice', 'Smith', 85); -- UPDATE existing rows UPDATE Students SET Grade = 90 WHERE StudentID = 101; -- DELETE rows (always use WHERE to avoid deleting everything!) DELETE FROM Students WHERE StudentID = 101;

Creating Tables with Constraints

SQL CREATE TABLE Students ( StudentID INTEGER PRIMARY KEY, FirstName VARCHAR(50) NOT NULL, LastName VARCHAR(50) NOT NULL, Grade INTEGER CHECK (Grade BETWEEN 0 AND 100), CourseID INTEGER, FOREIGN KEY (CourseID) REFERENCES Courses(CourseID) );
WHERE vs HAVING: WHERE filters individual rows before grouping. HAVING filters groups after GROUP BY. You cannot use aggregate functions in a WHERE clause — use HAVING instead.
Exam tip: Always alias table names in JOINs (e.g. Students s) and prefix columns with the alias when columns could be ambiguous. Cambridge mark schemes penalise ambiguous column references.

5 Database Management System (DBMS)

A Database Management System (DBMS) is software that provides a centralised interface for creating, querying, updating, and administering databases. Users and applications interact with the DBMS rather than directly with stored files.

Functions of a DBMS

FunctionDescription
Data storage managementManages how data is physically stored on disk; handles file organisation and indexing
Data retrievalProcesses SQL queries, optimises execution plans, returns results to the application
Data integrity enforcementApplies constraints (PK, FK, CHECK, NOT NULL) to prevent invalid data being stored
Access control / securityAuthenticates users; enforces permissions — which users can read, write, or modify which tables
Concurrency controlManages simultaneous access by multiple users to prevent data corruption (using locking or MVCC)
Transaction managementEnsures operations are atomic — either all changes commit or all are rolled back on failure
Backup and recoveryProvides tools to back up data and restore it after failure, using transaction logs
Data dictionary / schemaMaintains metadata: descriptions of tables, columns, types, constraints, and relationships

Data Dictionary

The data dictionary (system catalogue) is a repository of metadata — information about the data itself. It stores:

  • Table and column names with data types
  • Primary and foreign key definitions
  • Constraints and validation rules
  • User access permissions
  • Indexes and their associated columns

Concurrency Control

When multiple users update the same data simultaneously, problems arise without control:

  • Lost update: Two users read the same record (e.g. stock = 10). Both subtract 1 and write back 9. One update is lost — the correct result should be 8.
  • Dirty read: User A reads data that user B has modified but not yet committed. If B rolls back, A has read invalid data.
  • Phantom read: A query run twice returns different rows because another transaction inserted or deleted rows between the two reads.

Locking is the most common solution: before modifying a record, a transaction places a lock on it. Other transactions must wait until the lock is released. This serialises conflicting operations.

Deadlock: Two transactions each hold a lock the other needs — both wait forever. DBMSs detect deadlocks and automatically kill one transaction (which then rolls back) to allow the other to proceed.

6 ACID Properties & Transactions

A transaction is a logical unit of work — a sequence of operations that must all succeed together or all be undone. Example: transferring money from account A to B requires debiting A and crediting B atomically.

A
Atomicity
All operations in a transaction succeed completely, or none of them are applied. No partial updates.
C
Consistency
A transaction brings the database from one valid state to another. All integrity constraints remain satisfied.
I
Isolation
Concurrent transactions execute as if they were serial. Intermediate states are not visible to other transactions.
D
Durability
Once a transaction is committed, its changes persist even if the system crashes immediately afterwards.

COMMIT and ROLLBACK

SQL -- Start a transaction (some systems use BEGIN or START TRANSACTION) BEGIN TRANSACTION; -- Debit Account A UPDATE Accounts SET Balance = Balance - 100 WHERE AccountID = 1; -- Credit Account B UPDATE Accounts SET Balance = Balance + 100 WHERE AccountID = 2; -- If both succeed, commit permanently COMMIT; -- If anything fails (e.g. Account B does not exist), undo everything ROLLBACK;
Why ACID Matters — Bank Transfer Example

Suppose the system crashes after debiting Account A but before crediting Account B:

A
Atomicity: The debit is rolled back — neither account changes. Money is not lost.
C
Consistency: The total money across all accounts remains the same before and after — no constraint is violated.
I
Isolation: A third user querying Account A mid-transaction does not see the half-debited balance.
D
Durability: Once the COMMIT is confirmed, the transfer is permanent even if the power cuts out one second later.
Exam tip: When asked to explain ACID, always give a concrete example for each property. "Atomicity means all or nothing" is worth one mark; illustrating it with a bank transfer (debit + credit must both happen or neither) earns the application mark too.

📝 Exam Practice

Practice Question 1 — 4 marks

A library has Books and Members. A member can borrow many books, and the same book can be borrowed by many members over time. Describe, with an example, how this many-to-many relationship would be resolved in a relational database. State what attributes the link table should contain.

A many-to-many relationship cannot be directly implemented. A link table (e.g. Loan) is created with foreign keys referencing both parent tables:

  • Loan(LoanID, MemberID [FK → Member], BookID [FK → Book], LoanDate, ReturnDate)

MemberID and BookID are both foreign keys. Each row represents one borrowing event. Additional attributes of the relationship — such as LoanDate and ReturnDate — belong in the link table, not in Member or Book. A composite key (MemberID, BookID, LoanDate) could be used if a surrogate LoanID is not preferred.

Practice Question 2 — 6 marks

The following unnormalised table records student exam results:

StudentIDStudentNameTutorIDTutorNameSubjectCodeSubjectNameMark
S01AliceT1Mr HartCS01Computing88
S01AliceT1Mr HartMA01Maths72
S02BobT2Ms KayCS01Computing65

Normalise this table to 3NF. Show each stage and identify any dependencies removed.

1NF: Already in 1NF — atomic values, each row unique with composite PK (StudentID, SubjectCode).

2NF — remove partial dependencies:

  • StudentName, TutorID, TutorName depend only on StudentID (not the full composite PK) → partial dependency
  • SubjectName depends only on SubjectCode → partial dependency

After 2NF:

  • Student(StudentID, StudentName, TutorID)
  • Subject(SubjectCode, SubjectName)
  • Result(StudentID, SubjectCode, Mark)

3NF — remove transitive dependencies:

  • In the Student table: TutorName depends on TutorID, not on StudentID → transitive dependency

After 3NF:

  • Student(StudentID, StudentName, TutorID [FK])
  • Tutor(TutorID, TutorName)
  • Subject(SubjectCode, SubjectName)
  • Result(StudentID [FK], SubjectCode [FK], Mark)
Practice Question 3 — 5 marks

Using the tables Students(StudentID, FirstName, LastName, TutorGroupID) and TutorGroups(TutorGroupID, TutorName, Room), write SQL to:

  1. List the full name and tutor name for every student, sorted by last name
  2. Count the number of students in each tutor group, showing only groups with more than 15 students
SQL -- Part 1 SELECT s.FirstName, s.LastName, t.TutorName FROM Students s INNER JOIN TutorGroups t ON s.TutorGroupID = t.TutorGroupID ORDER BY s.LastName; -- Part 2 SELECT t.TutorName, COUNT(s.StudentID) AS StudentCount FROM Students s INNER JOIN TutorGroups t ON s.TutorGroupID = t.TutorGroupID GROUP BY t.TutorGroupID, t.TutorName HAVING COUNT(s.StudentID) > 15;
Practice Question 4 — 4 marks

Explain the four ACID properties of a database transaction. For each property, state what problem it prevents.

  • Atomicity — all operations in a transaction complete successfully, or none are applied. Prevents partial updates leaving the database in an inconsistent state (e.g. money debited but not credited).
  • Consistency — a transaction takes the database from one valid state to another, maintaining all defined constraints. Prevents constraint violations from being committed.
  • Isolation — concurrent transactions do not interfere with each other; each executes as if it were the only transaction running. Prevents dirty reads and lost updates caused by simultaneous access.
  • Durability — once committed, changes persist even after a system crash. Prevents data loss due to hardware or software failure after a successful commit.
Practice Question 5 — 3 marks

Explain what a primary key is, what a foreign key is, and how together they enforce referential integrity.

A primary key is an attribute (or set of attributes) that uniquely identifies each record in a table. It must be unique and cannot be NULL.

A foreign key is an attribute in one table whose values reference the primary key of another table, creating a link between the two tables.

Referential integrity is enforced by the DBMS: a foreign key value must either match an existing primary key value in the referenced table, or be NULL. This prevents orphan records — for example, an Enrolment row cannot reference a StudentID that does not exist in the Students table. If a student is deleted, the DBMS will either reject the deletion, cascade-delete their enrolments, or set the FK to NULL, depending on the rule configured.

ACID

Common Mistakes

  • Using WHERE instead of HAVING for aggregate conditions
  • Forgetting to check for partial deps in 2NF — only relevant when PK is composite
  • Putting relationship attributes in a parent table instead of the link table
  • Confusing INNER JOIN (matched rows only) with LEFT JOIN (all left rows)
  • Saying normalisation "speeds up queries" — it can actually slow reads; it removes anomalies
