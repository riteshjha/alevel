Further Hardware

RISC and CISC processor architectures, instruction pipelining and hazards, parallel and multi-core processing, cache memory levels and hit rates, and the full memory hierarchy — how modern hardware achieves high performance.

1 RISC vs CISC Architectures

Two competing design philosophies have shaped how processors are built. The debate goes back to the 1970s and continues to influence chip design today (ARM processors are RISC; x86 processors used in most PCs are CISC).

CISC — Complex Instruction Set Computer

The CISC approach provides a large instruction set where individual instructions can perform complex, multi-step operations (e.g., a single instruction can fetch a value from memory, add it to a register, and store the result). The philosophy is to make the assembly programmer's life easier and reduce code size.

Large Instruction Set

Hundreds of different instructions, many of which are rarely used. Reduces the number of instructions a programmer needs to write.

Variable Instruction Length

Instructions can be 1 to 15 bytes long (e.g., x86). Makes decoding more complex but reduces program memory usage.

Many Addressing Modes

A single instruction can reference memory in many different ways — immediate, direct, indirect, indexed, etc.

Complex Hardware

The control unit is large and complex (often microprogrammed). Transistors are used for decoding logic rather than registers.

Multiple Cycles

Many instructions take multiple clock cycles to complete. Performance gains come from the richness of individual instructions.

Examples

Intel x86, x86-64 (used in most desktop/laptop CPUs). Backwards compatibility has kept x86 CISC even as performance grew.

RISC — Reduced Instruction Set Computer

The RISC approach strips the instruction set down to a small number of simple instructions that each execute in exactly one clock cycle. The philosophy is that simpler hardware can be clocked faster and pipelined more effectively.

Small Instruction Set

Typically fewer than 100 instructions. Each instruction does one simple, well-defined operation.

Fixed Instruction Length

All instructions are the same number of bits (e.g., 32 bits in ARM). Makes fetch and decode stages fast and predictable.

Load/Store Architecture

Only dedicated LOAD and STORE instructions can access memory. All arithmetic is performed on values already in registers.

Large Register File

Many general-purpose registers (e.g., 32 in ARM). Reduces memory accesses — values stay in registers between operations.

One Cycle per Instruction

Designed so that each instruction completes in one clock cycle. Enables effective pipelining — the pipeline never stalls due to instruction complexity.

Examples

ARM (used in smartphones, tablets, Apple M-series chips), MIPS, RISC-V. Dominant in mobile and embedded systems.

Exam tip: A common question is "Explain two features of a RISC processor." Focus on: fixed-length instructions (enables fast decode), load/store architecture (all arithmetic on registers), and one clock cycle per instruction (enables efficient pipelining).
Practice Question

A software company is designing an embedded processor for a battery-powered IoT sensor. Explain, with reasons, whether a RISC or CISC architecture would be more suitable.

RISC is more suitable. (1 mark)

RISC processors have simpler hardware — fewer transistors needed for the control unit — which reduces power consumption, extending battery life. (1 mark)

Fixed-length instructions and a load/store architecture make RISC designs easier to pipeline efficiently, improving performance per watt. (1 mark)

The simpler design also generates less heat, which is important in a compact embedded device. (1 mark — accept any valid relevant reason)

2 Pipelining

Pipelining is a technique where the processor begins fetching the next instruction before it has finished executing the current one — similar to an assembly line in a factory. Each instruction passes through a series of stages, and different instructions occupy different stages simultaneously.

The Four-Stage Pipeline

The classic pipeline has four stages:

StageNameWhat Happens
FFetchThe instruction is fetched from memory using the Program Counter (PC); PC is incremented
DDecodeThe control unit decodes the instruction — determines the operation and identifies the operands
EExecuteThe ALU performs the operation (arithmetic, logic, comparison)
SStore / Write-backThe result is written back to a register or memory

Pipeline Diagram

Without pipelining, instruction 2 cannot start until instruction 1 is completely finished — only 1 instruction every 4 clock cycles. With pipelining, a new instruction enters the pipeline every cycle — throughput approaches 1 instruction per cycle.

Instr 2 —FDE S——
Instr 3 ——FD ES—
Instr 4 ———F DES

Without pipelining: 4 instructions × 4 cycles = 16 cycles. With pipelining: all 4 instructions complete by cycle 7 — just 7 cycles. For long programs the throughput approaches one instruction per cycle.

Pipeline Hazards

A pipeline hazard is any situation that prevents the next instruction from executing in its designated pipeline stage. Hazards cause pipeline stalls (also called bubbles) — wasted cycles where a stage does no useful work.

Data Hazard

Occurs when an instruction depends on the result of a previous instruction that has not yet finished. The second instruction must wait until the first writes its result.

Example: ADD R1, R2, R3 followed immediately by SUB R4, R1, R5 — SUB needs R1 but ADD hasn't written it yet.

Structural Hazard

Occurs when two instructions need the same hardware resource at the same time — for example, two instructions both trying to access memory in the same cycle (one to fetch, one to read data).

Control Hazard

Occurs after a branch instruction. The processor doesn't know which instruction to fetch next until the branch condition is evaluated. Instructions already in the pipeline may need to be discarded (flushed).

Handling Hazards

TechniqueDescriptionHazard addressed
Stalling (bubble insertion) The pipeline is paused — NOP (no-operation) instructions are inserted to create a gap while waiting for data Data hazard
Operand Forwarding (bypassing) The result of an instruction is passed directly to the next instruction's execute stage without waiting for a write-back — an extra internal data path bypasses the register file Data hazard
Out-of-Order Execution The processor reorders instructions so that independent instructions execute while waiting for a stalled instruction to complete Data hazard
Branch Prediction The processor predicts whether a branch will be taken and fetches instructions speculatively. If the prediction is wrong, the pipeline is flushed (speculative instructions are discarded) Control hazard
Delayed Branch The instruction immediately after a branch is always executed (delay slot). The compiler fills this slot with a useful instruction that doesn't depend on the branch outcome Control hazard
Worked Example — Identifying a Data Hazard

Consider these two instructions:

Instruction 1:  ADD R1, R2, R3    // R1 ← R2 + R3
Instruction 2:  MUL R4, R1, R5    // R4 ← R1 × R5

In a 4-stage pipeline, Instruction 1 writes R1 in cycle 4 (store stage). But Instruction 2 needs R1 in cycle 3 (execute stage, one cycle after Instruction 1 was fetched). This is a data hazard (RAW — Read After Write).

Resolution using stalling: Insert 1 NOP bubble between the two instructions, delaying Instruction 2 by one cycle so it reads R1 after it has been written.

Resolution using forwarding: Route the ALU output of Instruction 1 directly to the ALU input of Instruction 2 in the same cycle — no stall needed.

Practice Question

Explain what a control hazard is, and describe how branch prediction reduces its impact.

Control hazard: When the processor encounters a conditional branch instruction, it does not know which instruction to fetch next until the branch condition has been evaluated in the execute stage. Instructions already fetched into the pipeline may be wrong — they must be flushed (discarded), wasting cycles. (2 marks)

Branch prediction: The processor predicts whether the branch will be taken (based on past behaviour or static rules) and continues fetching instructions along the predicted path speculatively. If the prediction is correct, no cycles are wasted. If incorrect, the speculatively fetched instructions are flushed and the correct instructions are fetched — this still costs cycles, but far less often than if prediction were not used. (2 marks)

3 Parallel Processing

Pipelining achieves instruction-level parallelism within a single core. Parallel processing extends this idea by using multiple processing units to execute instructions or data operations simultaneously, dramatically increasing throughput for suitable workloads.

Flynn's Taxonomy

Flynn's taxonomy classifies computer architectures by the number of instruction streams and data streams they handle simultaneously:

SISD — Single Instruction, Single Data

The classic von Neumann model. One processor executes one instruction on one piece of data at a time. No parallelism.

Example: A simple 8-bit microcontroller.

SIMD — Single Instruction, Multiple Data

One instruction is broadcast to multiple processing units, each applying it to a different piece of data simultaneously. Ideal when the same operation is applied to large datasets.

Example: Adding 8 pairs of numbers in one cycle using vector/SSE registers. Used in graphics, scientific simulations, machine learning.

MISD — Multiple Instruction, Single Data

Multiple processors each execute different instructions on the same data. Rare in practice — used mainly in fault-tolerant systems where the same data is processed redundantly.

Example: Redundant flight control computers.

MIMD — Multiple Instruction, Multiple Data

Multiple processors each execute different instruction streams on different data simultaneously. The most general and widely used parallel model.

Example: Multi-core CPUs, server clusters, distributed computing systems.

Multi-Core Processors

A multi-core processor integrates two or more independent processor cores onto a single chip. Each core has its own pipeline, registers, and typically its own L1/L2 cache. The cores share L3 cache and main memory.

AdvantageLimitation
Multiple tasks can run truly in parallel — improved throughput and responsivenessNot all software is written to exploit multiple cores (serial code gets no speedup)
Better performance per watt than a single very fast coreSharing memory and cache can create bottlenecks (memory bandwidth)
One core can handle OS tasks while others run applicationsSynchronisation between cores (avoiding race conditions) is complex to program

GPU Parallel Processing

A GPU (Graphics Processing Unit) contains thousands of small, simpler cores designed for SIMD-style parallelism. While a CPU has 4–32 powerful cores optimised for sequential tasks, a GPU has thousands of smaller cores optimised for simultaneously processing many identical operations — originally for rendering pixels, now widely used for AI and scientific computing.

CPUGPU
Core count4–32 powerful coresThousands of simpler cores (e.g., 4096 CUDA cores)
Best forSequential, complex, branchy tasksHighly parallel, data-parallel tasks
Use casesOS, applications, databasesGraphics, AI training, scientific simulation

Amdahl's Law

Amdahl's Law states that the speedup from parallelisation is limited by the fraction of the program that must remain sequential. Even with an infinite number of processors, the maximum speedup is bounded by:

Max speedup = 1 / (1 − P)

where P is the fraction of the program that can be parallelised. If 90% of code can be parallelised (P = 0.9), the maximum speedup is 1/(1−0.9) = 10×, regardless of how many processors are added.

Key implication: Adding more cores gives diminishing returns. Doubling cores from 2 to 4 gives a bigger speedup than doubling from 100 to 200. The serial portion of code ultimately dominates performance.
Practice Question

Explain, using Flynn's taxonomy, why SIMD processing is well-suited to rendering a 3D scene in a video game.

SIMD (Single Instruction, Multiple Data) applies the same instruction to multiple data items simultaneously. (1 mark)

When rendering a 3D scene, the same transformation operations (e.g., applying a rotation matrix, computing lighting) must be applied independently to thousands or millions of pixels/vertices simultaneously. (1 mark)

SIMD allows one instruction (e.g., "multiply by this matrix") to be broadcast to many GPU cores, each processing a different pixel at the same time. This is far faster than applying the operation sequentially to each pixel. (1 mark)

4 Cache Memory

Cache memory is a small, very fast memory built close to (or directly on) the processor chip. It stores copies of the most recently or most frequently used instructions and data from main memory (RAM). Its purpose is to bridge the speed gap between the fast CPU and the comparatively slow RAM.

Why Cache is Needed

Modern CPUs can execute billions of instructions per second. Main memory (DRAM) has a latency of 50–100 nanoseconds — the CPU would spend most of its time waiting for data if it fetched everything from RAM. Cache memory (SRAM) has latency of 1–10 nanoseconds and can supply data fast enough to keep the CPU busy.

Cache Levels

Cache is organised into levels. Smaller and faster caches are closer to the processor core; larger and slower caches are further away:

Cache Hit and Miss

When the CPU requests data, it first checks the cache:

  • Cache hit — the requested data is in the cache. The CPU receives it quickly.
  • Cache miss — the data is not in the cache. It must be fetched from a lower (slower) level, causing a delay (stall).

The hit rate is the proportion of memory accesses that are served from the cache:

Hit rate = (Number of cache hits) ÷ (Total memory accesses)
Worked Example — Hit Rate Calculation

A program makes 10,000 memory accesses. 9,200 are served from cache; 800 require a fetch from RAM. Calculate the hit rate.

Hit rate = 9,200 ÷ 10,000 = 0.92 (92%)

A hit rate of 92% means the CPU avoids going to the slow RAM for 92% of accesses — a significant performance benefit.

Locality of Reference

Cache works because programs tend to access the same memory locations repeatedly — a property called locality of reference. There are two types:

TypeDefinitionWhy it helps cacheExample
Temporal locality A recently accessed memory location is likely to be accessed again soon Keeping recently used data in cache pays off — it will probably be needed again A loop variable is accessed on every iteration
Spatial locality If one memory location is accessed, nearby locations are likely to be accessed soon Cache loads a whole cache line (e.g., 64 bytes) at once — nearby data is brought in proactively Iterating through an array accesses consecutive addresses

Cache Replacement Policies

When the cache is full and a new block must be loaded, an existing entry must be evicted. Common policies include:

  • LRU (Least Recently Used) — evict the block that has not been accessed for the longest time. Exploits temporal locality. Most commonly used.
  • FIFO (First In, First Out) — evict the block that has been in the cache the longest, regardless of access recency.
  • Random replacement — evict a randomly chosen block. Simple to implement; performance close to LRU in practice.
Practice Question

Explain why iterating through a large array in sequential order results in a higher cache hit rate than accessing array elements in a random order.

When the first element of the array is accessed, the cache loads not just that element but the entire cache line containing it (typically 64 bytes = 8–16 integers). The following elements in the array are in adjacent memory locations — they are already in the cache when they are needed. This is spatial locality of reference. (2 marks)

In contrast, random access jumps to arbitrary locations. Each access is likely to be a cache miss because the data at that address was not recently used and the cache line loaded for the previous access does not contain the next element. The CPU must fetch from RAM each time, resulting in many slow cache misses. (2 marks)

5 Memory Hierarchy

No single memory technology is simultaneously fast, large, and cheap. The memory hierarchy organises different storage technologies into a pyramid — the fastest and most expensive storage is at the top (small capacity, close to the CPU), and the slowest and cheapest is at the bottom (large capacity, far from the CPU).

LevelTechnologySpeedCapacityCost/GBVolatile?
Registers Flip-flops inside CPU Fastest (~0.25 ns) Bytes (e.g., 32 × 64-bit) Extremely high Yes
L1 Cache SRAM on-core ~1 ns 16–64 KB Very high Yes
L2 Cache SRAM on-chip ~5 ns 128 KB – 2 MB High Yes
L3 Cache SRAM on-chip (shared) ~15–30 ns 4–64 MB High Yes
Main Memory (RAM) DRAM on motherboard ~50–100 ns 4 GB – 128 GB Moderate Yes
SSD Flash memory ~50–200 µs 256 GB – 4 TB Low No
HDD Magnetic rotating disk ~5–10 ms 500 GB – 20 TB Very low No

Key Trade-offs

  • Speed vs capacity: Faster memory has lower capacity. Registers are fastest but hold only a handful of values; an HDD is slowest but holds terabytes.
  • Speed vs cost: SRAM (cache) is far more expensive per bit than DRAM (RAM). Providing terabytes of cache would be prohibitively expensive.
  • Volatile vs non-volatile: All memories above secondary storage are volatile — their contents are lost when power is removed. SSDs and HDDs retain data without power.

Virtual Memory

Virtual memory extends the apparent size of RAM by using a portion of secondary storage (the swap file or page file) as if it were additional RAM. When RAM is full, the operating system moves pages (fixed-size blocks) of data that have not been used recently from RAM to disk.

  • Benefit: Programs can use more memory than is physically installed.
  • Cost: Accessing the swap file on disk is thousands of times slower than accessing RAM — called a page fault. Excessive swapping (thrashing) causes severe slowdowns.
Exam tip: If asked "why does a computer with virtual memory run slowly when many programs are open," the answer is thrashing — the OS spends more time swapping pages between RAM and disk than executing instructions. Each page fault is a slow disk access.
Practice Question

Explain why it is not practical to build a computer that uses only register-speed memory for all storage, and describe how the memory hierarchy addresses this problem.

Why not all register-speed: SRAM (the technology used for registers and cache) is extremely expensive and physically large. Providing even 1 GB of register-speed memory would be prohibitively costly and would not fit on a chip. (1 mark)

How the hierarchy helps: The hierarchy places a small amount of very fast, expensive memory (registers, cache) close to the CPU, and uses progressively larger but slower and cheaper technologies for the levels below. (1 mark)

Cache exploits locality of reference — it automatically stores the most recently/frequently used data, so the CPU accesses fast cache most of the time and only accesses slow RAM on a cache miss. This gives performance close to cache speed at the cost of RAM. (1 mark)

The hierarchy as a whole provides the illusion of large, fast, cheap memory by ensuring that active data migrates to the higher levels automatically. (1 mark)

📝 Exam Practice

Question 1 — 6 marks

Compare RISC and CISC processor architectures. Your answer should refer to instruction set, instruction length, number of registers, and suitability for pipelining.

Instruction set: RISC has a small, simple instruction set (typically <100 instructions); CISC has a large, complex set (hundreds of instructions, including complex multi-step operations). (1 mark)

Instruction length: RISC uses fixed-length instructions (e.g., all 32 bits); CISC uses variable-length instructions (e.g., 1–15 bytes in x86). Fixed length makes fetch and decode faster. (1 mark)

Registers: RISC has many general-purpose registers (e.g., 32), reducing memory accesses; CISC has fewer registers (e.g., 8 in original x86). (1 mark)

Pipelining: RISC is better suited — each instruction takes one cycle and is the same length, so the pipeline stages are always the same duration and no stalls occur due to instruction complexity. CISC instructions take variable numbers of cycles, making pipeline stages uneven and harder to implement. (2 marks)

Memory access: RISC uses a load/store model — only LOAD/STORE access memory; CISC allows any instruction to access memory directly. (1 mark)

Question 2 — 4 marks

The following two instructions are executed in a pipelined processor:

Instruction 1:  ADD R3, R1, R2    // R3 ← R1 + R2
Instruction 2:  SUB R5, R3, R4    // R5 ← R3 − R4

(a) Identify the type of hazard that occurs. (1 mark)

(b) Describe two techniques the processor could use to resolve this hazard. (3 marks)

(a) Data hazard (RAW — Read After Write). Instruction 2 reads R3 before Instruction 1 has written the result of the ADD to R3. (1 mark)

(b) Any two of:

  • Stalling (bubble insertion): A NOP instruction is inserted between the two instructions, pausing Instruction 2 by one cycle until R3 has been written. (1 mark) This wastes a pipeline cycle but ensures correctness. (1 mark)
  • Operand forwarding (bypassing): The result from the ALU output of Instruction 1 is routed directly to the ALU input of Instruction 2 without waiting for a write-back to the register file. (1 mark) No pipeline stall is needed. (1 mark)
  • Out-of-order execution: An independent instruction that does not depend on R3 is moved between Instructions 1 and 2, allowing Instruction 1 time to write R3 while useful work is done. (1 mark)
Question 3 — 4 marks

A processor has an L1 cache, an L2 cache, and main memory (RAM). Describe what happens when the processor needs to read a value and it is not in either cache level. Include in your answer what happens to the cache after the value is retrieved.

The processor first checks L1 cache — the data is not there (L1 miss). (1 mark)

It then checks L2 cache — the data is also not there (L2 miss). (1 mark)

The processor sends a request to main memory (RAM). This causes a pipeline stall while waiting for RAM to respond (RAM is much slower — typically 50–100 ns vs ~1 ns for L1). (1 mark)

When the data arrives from RAM, it is written into both the L1 and L2 caches (as a cache line — a block of adjacent memory). The requested value is forwarded to the processor. Future accesses to the same or nearby locations will now find the data in cache (a hit), exploiting spatial and temporal locality. (1 mark)

Question 4 — 3 marks

A processor executes 50,000 memory accesses. The L1 cache has a hit rate of 85%. Accesses that miss L1 are served by L2, which has a hit rate of 90% for those remaining accesses. The rest go to RAM.

Calculate: (a) the number of L1 hits, (b) the number of L2 hits, (c) the number of RAM accesses.

(a) L1 hits: 50,000 × 0.85 = 42,500 (1 mark)

L1 misses: 50,000 − 42,500 = 7,500

(b) L2 hits: 7,500 × 0.90 = 6,750 (1 mark)

(c) RAM accesses: 7,500 − 6,750 = 750 (1 mark)

Question 5 — 4 marks

Explain, using Flynn's taxonomy, the difference between SIMD and MIMD processing. Give one example of a use case for each.

SIMD (Single Instruction, Multiple Data): One instruction stream is applied simultaneously to multiple data items. All processing units execute the same operation on different pieces of data at the same time. (1 mark) Example: Applying a brightness filter to every pixel in an image simultaneously using a GPU. (1 mark)

MIMD (Multiple Instruction, Multiple Data): Multiple independent processors each execute their own instruction stream on their own data simultaneously. Each core operates independently. (1 mark) Example: A quad-core CPU where core 1 runs a browser, core 2 runs a video encoder, core 3 runs an antivirus scan, and core 4 handles OS tasks — all at the same time. (1 mark)

Question 6 — 3 marks

Explain what is meant by thrashing in the context of virtual memory, and describe one consequence for the user.

Thrashing occurs when the computer is running more processes than its physical RAM can comfortably hold. (1 mark)

The operating system constantly swaps pages between RAM and the swap file on disk (page fault after page fault), spending more time moving data than executing program instructions. (1 mark)

Consequence: The computer becomes extremely slow and unresponsive — the user experiences severe lag or the system appears almost frozen, because disk access (SSD/HDD) is thousands of times slower than RAM access. (1 mark)

Common Mistakes

  • RISC means fewer instructions, not a slower processor — fewer, simpler instructions enable faster clocks
  • A pipeline hazard slows throughput but does not mean the pipeline is broken — it inserts a stall (bubble)
  • Cache miss costs depend on which level misses — an L1 miss that hits L2 is much cheaper than a full RAM fetch
  • Virtual memory is not RAM — it uses disk, which is thousands of times slower
  • Amdahl's Law: adding more cores only helps the parallel fraction — the serial fraction is the bottleneck
