Further Data Representation

Binary floating-point representation, normalisation, range vs precision trade-offs, rounding and truncation errors, overflow and underflow — the mathematics of how real numbers are stored in a computer.

Paper 3 · A Level Only ~12% of Paper 3 marks

  • Convert a denary real number into binary floating-point form
  • Convert a binary floating-point number back to its denary value
  • Normalise a positive and negative binary floating-point number
  • Explain what normalisation achieves and why it is used
  • Explain the effect of increasing the number of bits allocated to the exponent vs the mantissa
  • Describe the effect of rounding and truncation on a stored value
  • Define and give an example of overflow and underflow
  • Represent positive and negative integers using two's complement

1 Floating-Point Representation

Integers in binary are straightforward — each bit represents a power of 2. But real numbers (numbers with fractional parts, or very large/small numbers) cannot be stored efficiently as simple integers. Floating-point representation solves this by separating a number into two parts: the significant digits and the scale.

The idea is analogous to scientific notation in decimal: the number 0.000345 is written as 3.45 × 10−4. The mantissa (3.45) holds the significant digits; the exponent (−4) sets the scale.

Structure of a Floating-Point Number

A binary floating-point number is stored in three fields:

S
Exponent (n bits)
Mantissa (m bits)
Sign bit Exponent Mantissa
FieldPurposeNotes
Sign bit Indicates positive (0) or negative (1) The most significant bit of the mantissa — the sign of the whole number
Mantissa Stores the significant digits of the number More mantissa bits → more precision (more significant figures)
Exponent Stores the power of 2 that scales the mantissa More exponent bits → greater range (larger and smaller numbers representable); stored in two's complement

The Formula

The value of a floating-point number is:

Value = Mantissa × 2Exponent

The mantissa is treated as a binary fraction — the binary point is assumed to be immediately after the sign bit. So a mantissa of 0.10110 represents the value ½ + ⅛ + 1/16 = 0.6875 in the positive case.

Cambridge convention: In Cambridge 9618, the mantissa is a two's complement fraction with the binary point after the sign (leftmost) bit. The exponent is a two's complement integer. Always check your exam question for the exact format and number of bits specified.

2 Converting Between Denary and Floating-Point Binary

Denary → Binary Floating Point

The general method is:

  1. Convert the denary number to binary (integer and fractional parts separately).
  2. Write in the form 0.1… (positive) or 1.0… (negative) — this is the normalised form.
  3. Count how many places the binary point moved to reach the normalised form — this is the exponent.
  4. Store the mantissa bits and the exponent (in two's complement) in the allocated fields.
Worked Example — Denary to Floating Point (positive)

Represent +6.5 in an 8-bit format: 5-bit mantissa, 3-bit exponent.

1Convert 6.5 to binary: 6 = 110₂, 0.5 = 0.1₂ → 110.1
2Write in normalised form: move binary point 3 places left → 0.1101 × 23
(normalised positive: starts with 0.1)
3Mantissa = 01101 (5 bits, binary point after sign bit — the 0. prefix is implicit)
4Exponent = +3 = 011 in 3-bit two's complement
5Final stored bit pattern: 01101 011
Worked Example — Denary to Floating Point (negative)

Represent −6.5 in an 8-bit format: 5-bit mantissa, 3-bit exponent.

1Convert +6.5 to binary: 110.1
2Convert to two's complement fraction. First write 0.1101, then negate:
Invert all bits: 1.0010, add 1 to last bit: 1.0011
(normalised negative: starts with 1.0)
3Mantissa = 10011
4Exponent = +3 = 011 (same scale, only the mantissa is negated)
5Final stored bit pattern: 10011 011

Binary Floating Point → Denary

  1. Read the exponent (two's complement integer) — call it e.
  2. Read the mantissa (two's complement fraction with binary point after sign bit).
  3. Shift the binary point of the mantissa e places to the right (positive e) or left (negative e).
  4. Convert the resulting binary integer/fraction to denary.
Worked Example — Binary Floating Point to Denary

Convert the 8-bit value 01011 010 (5-bit mantissa, 3-bit exponent) to denary.

1Mantissa = 01011 → as a fraction: 0.1011
= ½ + ⅛ + 1/16 = 0.5 + 0.125 + 0.0625 = 0.6875
2Exponent = 010 = +2 (two's complement)
3Value = 0.6875 × 2² = 0.6875 × 4 = 2.75
Worked Example — Negative Exponent

Convert 01100 110 (5-bit mantissa, 3-bit exponent) to denary.

1Mantissa = 01100 → 0.1100 = ½ + ¼ = 0.75
2Exponent = 110 in two's complement:
MSB is 1 → negative. Invert: 001, add 1: 010 = 2 → exponent = −2
3Value = 0.75 × 2−2 = 0.75 ÷ 4 = 0.1875

3 Normalisation

Normalisation is the process of adjusting a floating-point number so that the mantissa uses all available bits to represent significant digits — maximising precision for a given number of bits.

Rules for a Normalised Mantissa

A floating-point number (with the binary point after the sign bit) is normalised when:

Sign of NumberNormalised Form Starts WithMeaning
Positive (sign bit = 0) 0.1… The bit immediately after the binary point must be 1
Negative (sign bit = 1) 1.0… The bit immediately after the binary point must be 0

If a mantissa has leading zeros after the sign bit (positive) or leading ones (negative), it is not normalised and precision is being wasted.

Why normalise? If the mantissa were 0.001011 instead of 0.101100, the first two bits after the point carry no information — they are just padding. Normalisation shifts those significant bits into the high-order positions, maximising the precision representable with the given number of mantissa bits.

How to Normalise

To normalise an un-normalised number:

  1. Shift the mantissa left until it starts with 0.1 (positive) or 1.0 (negative).
  2. For each left-shift of the mantissa, decrease the exponent by 1.
  3. Truncate or round the mantissa to fit in the allocated bits.
Worked Example — Normalising a Positive Number

Normalise the mantissa 0.001011 with exponent +5 (6-bit mantissa).

1Mantissa starts 0.00… — not normalised. Shift left 2 places:
0.001011 → shift 1: 0.01011 (exp → 4)
→ shift 2: 0.1011 (exp → 3) — now starts with 0.1 ✓
2Normalised mantissa: 0.101100 (padded to 6 bits with trailing zero)
3New exponent: +3

Value check: 0.101100 × 2³ = 0.6875 × 8 = 5.5 ✓ (same value, more precision used)

Worked Example — Normalising a Negative Number

Normalise the two's complement mantissa 1.11001 with exponent +2 (6-bit mantissa).

1Mantissa starts 1.1… — not normalised (normalised negative starts 1.0…). Shift left 1 place:
1.11001 → 1.10010 (exp → 1) — still starts 1.1…
→ shift again: 1.00100 (exp → 0) — starts 1.0… ✓
2Normalised mantissa: 1.00100
3New exponent: 0

Detecting an Unnormalised Number

You can quickly check if a number is normalised by looking at just the first two bits of the mantissa:

  • 01… — normalised positive ✓
  • 10… — normalised negative ✓
  • 00… — NOT normalised (positive with wasted leading zero)
  • 11… — NOT normalised (negative with wasted leading one)
Practice Question

Normalised mantissa: 0.101100, exponent: +3

(b) Starts 1.1… — NOT normalised. Shift left 1:

Normalised mantissa: 1.00110, exponent: −2

4 Range vs Precision

A floating-point format has a fixed total number of bits. Those bits must be divided between the exponent and the mantissa. This is a trade-off: you cannot increase both range and precision simultaneously.

More Exponent Bits

Increases the range of numbers that can be represented — very large and very small numbers become possible. But fewer bits remain for the mantissa, reducing precision.

More Mantissa Bits

Increases precision — more significant digits can be stored, reducing rounding errors. But fewer bits remain for the exponent, reducing range.

Range

The span of values that can be represented, from the most negative to the most positive number. Determined primarily by the number of exponent bits.

Precision

The number of significant digits that can be accurately stored. More mantissa bits = finer granularity between adjacent representable numbers.

Effect of Changing the Number of Bits

ChangeEffect on RangeEffect on Precision
Increase exponent bits (reduce mantissa bits)Increases — larger/smaller numbers possibleDecreases — fewer significant figures
Increase mantissa bits (reduce exponent bits)Decreases — fewer extreme valuesIncreases — more significant figures
Increase total bits (both)IncreasesIncreases
Exam tip: A common question is "A programmer changes a floating-point format from 8 bits to 16 bits. State the effect on range and precision." Answer: Both range and precision increase because more bits are available for both fields. If the question says "the exponent is kept the same size but the total increases," only precision increases.

The IEEE 754 Standard

Real processors use the IEEE 754 standard for floating-point arithmetic. Cambridge 9618 does not require you to memorise the full IEEE 754 format, but you should know its key features:

  • Single precision: 32 bits total — 1 sign bit, 8 exponent bits, 23 mantissa bits
  • Double precision: 64 bits total — 1 sign bit, 11 exponent bits, 52 mantissa bits
  • The exponent is stored as a biased (offset) integer, not two's complement (the bias is 127 for single precision)
  • Special values: +∞, −∞, NaN (Not a Number) are encoded using reserved exponent patterns
  • Double precision has greater range and precision than single precision — used in scientific computing
Single Precision (32-bit)Double Precision (64-bit)
Sign bits11
Exponent bits811
Mantissa bits2352

Max positive value~3.4 × 1038~1.8 × 10308

5 Errors in Floating-Point Representation

Because a floating-point number uses a finite number of bits, not every real number can be represented exactly. This leads to several types of error.

Rounding Error

When a number cannot be represented exactly in the available mantissa bits, the stored value is rounded to the nearest representable value. The difference between the true value and the stored value is a rounding error.

Example

In decimal: 1 ÷ 3 = 0.333333… The stored value (e.g., 0.3333) differs from the true value by 0.0000333… This tiny difference can accumulate over many calculations — known as accumulation of rounding errors.

In binary: the decimal fraction 0.1 has no exact binary representation (0.000110011… repeating). It is stored as an approximation, which is why 0.1 + 0.2 ≠ 0.3 in most programming languages.

Truncation Error

Instead of rounding, some systems simply discard (truncate) the extra bits that don't fit. This always produces an error in the same direction (the stored value is always smaller in magnitude than the true value for positive numbers).

RoundingTruncation
MethodRound to nearest representable valueDiscard all bits beyond mantissa size
Error directionCan go up or downAlways in same direction (magnitude reduced)
Max error½ of least significant bit positionUp to 1 least significant bit

Absolute and Relative Error

  • Absolute error = |true value − stored value|
  • Relative error = absolute error ÷ |true value|

Relative error is often more meaningful — a rounding error of 0.001 is tiny when the true value is 1000, but enormous when the true value is 0.001.

Overflow

Overflow occurs when the result of a calculation is too large (in magnitude) to be stored in the available exponent bits. The exponent required exceeds the maximum representable exponent.

Effect: The result cannot be stored — typically the system produces a special value (∞ in IEEE 754) or an error flag. In exam context: state that the result is too large to be represented in the format and the stored value will be incorrect / the program may crash.

Underflow

Underflow occurs when the result of a calculation is too small (too close to zero) to be represented — the exponent required is smaller than the minimum representable exponent. The value is effectively rounded down to zero.

OverflowUnderflow
What happensResult too large — exponent too bigResult too small — exponent too negative
EffectCannot store — infinity / errorResult rounded to zero / error
Example10500 in single precision10−500 in single precision

Accumulation of Errors

In a long chain of arithmetic operations, rounding errors can accumulate. A small error at each step can grow to a significant error in the final result. This is particularly dangerous in:

  • Iterative numerical methods (e.g., solving differential equations)
  • Subtraction of two nearly equal numbers (catastrophic cancellation)
  • Financial calculations (which is why dedicated decimal types are used instead of floating point in finance)
Practice Question

A programmer uses a 6-bit mantissa floating-point format. Explain why the denary value 0.1 cannot be stored exactly, and describe the type of error that results.

0.1 in binary is 0.000110011… (repeating). It has an infinite binary expansion and therefore cannot be stored exactly in a finite number of mantissa bits. When stored in a 6-bit mantissa, the remaining bits are discarded — this is a truncation error (or rounding error, depending on the method used). The stored value is only an approximation of 0.1, and repeated use of this approximation in arithmetic can cause accumulation of errors.

Practice Question

A floating-point number system uses a 4-bit exponent stored in two's complement. The maximum positive exponent is +7 and the minimum is −8. Explain what happens if the result of a multiplication gives an exponent of +9.

The required exponent (+9) exceeds the maximum representable exponent (+7) for this format. This is overflow. The result cannot be stored — the stored value will be incorrect (possibly set to the maximum representable value, zero, or a special error value, depending on the system). Any subsequent calculations using this incorrect value will also produce wrong results.

6 Two's Complement (Review & Extension)

Two's complement is the standard way to represent signed integers in binary — and it is used for both the mantissa (as a fraction) and the exponent (as an integer) in floating-point numbers.

Representing Integers in Two's Complement

For an n-bit two's complement integer:

  • The most significant bit (MSB) has a place value of −2n−1 (negative weight).
  • All other bits have their usual positive place values.
  • Range: −2n−1 to +2n−1 − 1
BitsRangeExample: −5
4-bit−8 to +71011 (−8 + 2 + 1 = −5)
8-bit−128 to +12711111011

Converting a Negative Number to Two's Complement

To represent −n in two's complement:

  1. Write the positive value n in binary.
  2. Invert all bits (flip 0s to 1s and 1s to 0s).
  3. Add 1 to the result.
Worked Example — Represent −19 in 8-bit Two's Complement
1+19 in 8-bit binary: 00010011
2Invert all bits: 11101100
3Add 1: 11101101

Check: −128 + 64 + 32 + 8 + 4 + 1 = −128 + 109 = −19 ✓

Addition in Two's Complement

Two's complement addition works just like normal binary addition. Any carry out of the MSB is discarded. The result is correct as long as overflow does not occur.

Worked Example — 25 + (−19) in 8-bit Two's Complement

+25 = 00011001    −19 = 11101101

  00011001  (+25)
+ 11101101  (−19)
──────────
1 00000110  (carry out discarded)
= 00000110 = +6 ✓

Subtraction Using Two's Complement

To compute A − B using two's complement: convert B to its two's complement (negate it) and then add. This is why computers only need an adder circuit — subtraction is just addition of the negated value.

Worked Example — 12 − 30 in 8-bit Two's Complement

12 − 30 = 12 + (−30). First find −30:

+30 = 00011110 → invert: 11100001 → add 1: 11100010 = −30

  00001100  (+12)
+ 11100010  (−30)
──────────
  11101110  = −18 ✓

Check: −128 + 64 + 32 + 8 + 4 + 2 = −128 + 110 = −18 ✓

Overflow in Two's Complement Addition

Overflow occurs when adding two numbers of the same sign produces a result with the opposite sign — the result exceeds the representable range.

  • Adding two positives → result is negative: overflow
  • Adding two negatives → result is positive: overflow
  • Adding one positive and one negative: never overflows
Practice Question

Using 8-bit two's complement, calculate −85 + (−50). State whether overflow occurs and explain your reasoning.

−85: +85 = 01010101 → invert: 10101010 → +1: 10101011

−50: +50 = 00110010 → invert: 11001101 → +1: 11001110

  10101011  (−85)
+ 11001110  (−50)
──────────
1 01111001  (carry discarded)
= 01111001 = +121

Overflow has occurred. We added two negative numbers but got a positive result (+121). The true result is −135, which is outside the 8-bit two's complement range (−128 to +127). The carry out of the MSB is the symptom, but the real indicator is that two negative operands produced a positive result.

The mantissa of a floating-point number is 0.001011. State whether this mantissa is normalised. If not, write the normalised form and state the adjustment needed to the exponent.

Question 4 — 4 marks

A programmer is choosing between two floating-point formats, both using 16 bits total:

  • Format A: 10-bit mantissa, 6-bit exponent
  • Format B: 6-bit mantissa, 10-bit exponent

Compare the range and precision of the two formats. State which format would be more appropriate for a scientific simulation that involves very large and very small numbers, with only moderate accuracy needed.

Format A (10-bit mantissa, 6-bit exponent): Greater precision — more significant figures stored. Smaller range — fewer extreme values representable. (1 mark)

Format B (6-bit mantissa, 10-bit exponent): Greater range — can represent very large and very small numbers. Less precision — fewer significant figures. (1 mark)

Recommendation: Format B is more appropriate for the simulation. (1 mark) Because the numbers involved are very large and very small (requiring a wide range), and only moderate accuracy is needed (so the reduced precision of 6-bit mantissa is acceptable). (1 mark)

Question 5 — 2 marks

Explain the difference between overflow and underflow in the context of floating-point representation.

Overflow: The result of a calculation is too large in magnitude to be represented — the required exponent exceeds the maximum that can be stored. The result cannot be stored correctly (1 mark).

Underflow: The result of a calculation is too small in magnitude (too close to zero) to be represented — the required exponent is smaller (more negative) than the minimum that can be stored. The result is rounded to zero (1 mark).

Common Mistakes

  • Forgetting the binary point is after the sign bit, not before all bits
  • Treating the exponent as unsigned — it uses two's complement
  • Normalising positive numbers to 1.0… — that is the negative normalised form
  • Confusing overflow (too large) with underflow (too close to zero)
  • When normalising, forgetting to update the exponent for every shift applied
